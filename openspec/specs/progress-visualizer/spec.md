# 仕様書: 進捗ビジュアライザー (Progress Visualizer)

## Purpose
ユーザーのファーミング進捗を可視化し、過去のスナップショットとの比較を通じて達成感をフィードバックする機能。ソルバー実行成功後にモーダルとして表示される。
## Requirements
### Requirement: 進捗報告画面の表示
システムは、ユーザーの育成進捗を報告するマシュ進捗パネルを、ダッシュボードの「あなたの育成進捗」セクション（達成率円グラフの上）に表示しなければならない (SHALL)。比較対象が無い場合も破綻せず案内を表示しなければならない (SHALL)。

#### Scenario: ダッシュボードでの進捗表示
- **WHEN** ダッシュボードを開き、現在の育成状態と進捗データが取得できたとき
- **THEN** 「あなたの育成進捗」セクション上部にマシュ進捗パネルが表示され、下部に既存の達成率円グラフが表示される。

#### Scenario: 比較対象なし時の表示
- **WHEN** 過去スナップショットが存在しないか、指定期間のスナップショットが取得できないとき
- **THEN** 「登録お疲れ様」モードのメッセージとともにパネルが表示される。

### Requirement: 進捗 tier の判定
システムは、進捗の達成度を `legendary` / `large` / `medium` / `small` / `none` の 5 段階（tier）で判定しなければならない (SHALL)。主指標は「バッファ込み実効不足の解消ぶんの推定周回数（前進周回）」を経過日数でならした値（周/日）とする。

前進周回は次で算出しなければならない (SHALL): 素材ごとに 単価（1個あたり推定周回数）= 1 ÷ ユーザー選択クエスト内の最高ドロップ率（選択クエスト内にドロップが無い素材は全クエスト中の最高率にフォールバック）、実効目標 = 育成目標の必要数 +（ストックバッファが有効なら バッファ量、無効なら 0）（quest-efficiency 機能の実効必要数と同一の定義）、現在所持は過去所持で下限クランプ（消費中立）した上で、実効不足の過去値と現在値の差分に単価を掛けて合算する。入手経路（周回・イベント交換・ストーリー報酬）は問わない。

しきい値（周/日）: `≥100 → legendary`、`≥30 → large`、`≥15 → medium`、`>0 → small`、`0 → none`（補完判定へ）。前進周回が 0 の期間は、労力周回（獲得素材の周回換算）に同じしきい値を適用して tier を補完しなければならない (SHALL)。ただし補完による tier は `large` を上限とし、`legendary` は前進周回によってのみ到達できる (SHALL)。

#### Scenario: 周回換算による判定
- **WHEN** 前進周回の周/日が算出できたとき
- **THEN** `≥100 → legendary`、`≥30 → large`、`≥15 → medium`、`>0 → small` と判定する。

#### Scenario: バッファ込み実効不足への前進
- **WHEN** 育成目標は充足済みだが、所持がストックバッファ込みの実効目標に満たない素材を獲得したとき
- **THEN** その獲得分も前進周回に計上する（ストックバッファ機能が無効なユーザーはバッファ0として従来の育成目標のみで判定する）。

#### Scenario: 育成消費が tier を下げないこと
- **WHEN** 同一期間内に素材の獲得と育成による消費の両方が発生したとき
- **THEN** 消費は前進周回を減少させず、獲得のみが tier 判定に寄与する。

#### Scenario: 選択クエストにドロップが無い素材の単価
- **WHEN** 不足解消した素材のドロップ元がユーザーの選択クエストに含まれないとき
- **THEN** 全クエスト中の最高ドロップ率で単価を算出し、0 扱いにしない。

#### Scenario: 前進ゼロ時のスループット補完
- **WHEN** 前進周回が 0 で、獲得素材（余剰込み）の労力周回が正のとき
- **THEN** 労力周回に同じしきい値を適用して tier を補完する。ただし `legendary` には到達させない（上限 `large`）。

#### Scenario: 進捗ゼロの判定
- **WHEN** 前進周回・労力周回・育成総量・新規入手のいずれも無いとき
- **THEN** tier は `none`、フォールバックは進捗ゼロと判定する。

### Requirement: rarity 別の必要AP推定値の事前計算
システムは、各 rarity（1〜5）から代表サーヴァントを抽出して solver で必要APを計算し、その平均値（または中央値）を保持しなければならない (SHALL)。

#### Scenario: rarity-AP テーブルの生成
- **WHEN** 事前計算プロセスが実行されたとき
- **THEN** 各 rarity からランダムに複数体のサーヴァントを選択し、共通目標（再臨/スキル/アペンド）達成までの必要APを solver で算出する。
- **THEN** 結果の平均値または中央値を rarity-AP テーブルとして保持する。

#### Scenario: 高難易度アクセス考慮の2セット計算
- **WHEN** rarity-AP テーブルを生成するとき
- **THEN** 「高難易度なし」と「高難易度あり（オーディールコール、冠位戦含む）」の 2 セットを生成する。

### Requirement: 高難易度コンテンツアクセス可否の判定
システムは、ユーザーの選択クエストリストから高難易度コンテンツ（オーディールコール、冠位戦）への到達可否を推測しなければならない (SHALL)。

#### Scenario: アクセス可否の推測
- **WHEN** 進捗計算を行うために rarity-AP テーブルを選択する必要があるとき
- **THEN** ユーザーが選択したクエストに該当コンテンツの代表クエストIDが含まれるかをチェックする。
- **THEN** 含まれる場合は「高難易度あり」テーブル、含まれない場合は「基本」テーブルを使用する。

### Requirement: この期間の活動の表示（獲得素材・スキル合計・育成総量）
システムは、比較スナップショット以降の育成活動を「この期間の活動」として表示しなければならない (SHALL)。指標は (a) 獲得した素材の合計個数（QP除外）、(b) 所持サーヴァントのスキル現在レベル合計の増分（新規入手サーヴァントを含む）、(c) 育成目標レンジ（再臨/スキル/アペンド）が縮んだ合計（育成総量）であり、いずれも値が正のものだけ表示する。

#### Scenario: スキル合計の変化の表示
- **WHEN** 所持サーヴァント（新規入手を含む）のスキル現在レベル合計が比較スナップショットから増えたとき
- **THEN** その増分を「スキル合計 +N」として表示する。比較対象に存在しない新規サーヴァントのスキルも増分として数える。

#### Scenario: 育成総量の表示
- **WHEN** 比較スナップショット間でサーヴァントの育成目標レンジ合計が縮んだとき
- **THEN** 縮んだ量の合計を「育成総量」として表示する。

### Requirement: 素材スループットの算出
システムは、比較スナップショット以降の所持数の増減から素材スループットを算出しなければならない (SHALL)。所持が純増した素材の合計個数を「獲得素材」、純減した合計個数を「育成投入」とし、QP（他素材と比べ桁違いに所持数が多いため）は除外する。

さらに、獲得素材を素材ごとの単価（1個あたり推定周回数）で換算した合計を「労力周回（推定活動量）」として算出しなければならない (SHALL)。労力周回は余剰在庫行きの獲得も含み、育成投入は含まない（育成の認知は育成総量・サーヴァント成長表示が担う）。労力周回は (a) 「この期間の活動」での「推定活動量 約N周相当」表示、(b) 前進ゼロ時の tier 補完、(c) マシュメッセージの2軸修飾に用いる。

#### Scenario: QP の除外
- **WHEN** 所持差分に QP の増減が含まれるとき
- **THEN** QP の増減はスループット・労力周回に含めない。

#### Scenario: 労力周回の表示
- **WHEN** 労力周回が正のとき
- **THEN** 「この期間の活動」に「推定活動量 約N周相当」を表示する。

#### Scenario: 育成投入は労力に含めない
- **WHEN** 所持が純減した素材（育成投入）があるとき
- **THEN** その換算周回は労力周回に加算しない。

### Requirement: マシュによる進捗フィードバック
進捗報告画面において、マシュ・キリエライトの画像と進捗種別に応じたメッセージを表示しなければならない (SHALL)。メッセージは方向性（tier）と労力（労力周回）の2軸で選択・修飾しなければならない (SHALL)。

#### Scenario: tier 別マシュメッセージ
- **WHEN** 進捗 tier が `legendary` / `large` / `medium` / `small` / `none` のいずれかに判定されたとき
- **THEN** その tier に対応する事前定義済みマシュメッセージ群からランダムにメッセージが選択され、マシュの画像とともに表示される。`legendary` はボックス/レイド全力月に相応しい最大級の称賛トーンとする。

#### Scenario: 労力による修飾
- **WHEN** tier が `medium` 以下で、労力周回が `large` 相当以上のとき
- **THEN** 活動量・備蓄を労うトーンのメッセージを優先する（「たくさん動いた月」を認知する）。

#### Scenario: 「お疲れ様」モード
- **WHEN** 過去スナップショットがない、または全指標で改善ゼロのとき
- **THEN** マシュの「登録お疲れ様」系メッセージが表示される。

#### Scenario: 新規入手・育成は tier に依らず労う
- **WHEN** 新規入手サーヴァントが居る、または育成総量が正のとき
- **THEN** tier が `none` であっても「休んでください」系ではなく、新規入手の歓迎／育成を労うメッセージを優先して表示する。

### Requirement: 達成感の視覚的演出
進捗 tier（5段階）に応じて、パネル内の色・サイズ・装飾の強度を変化させなければならない (SHALL)。`legendary` は `large` を上回る特別な演出とする。アニメーションは v1 では実装しない。

#### Scenario: legendary のビジュアル
- **WHEN** tier が `legendary` のとき
- **THEN** `large` よりさらに強い特別な配色・装飾で最大級の達成感を演出する。

#### Scenario: tier 別ビジュアル
- **WHEN** tier が `large` のとき
- **THEN** 派手な配色と大きい数値表示で達成感を強調する。

#### Scenario: 控えめな表示
- **WHEN** tier が `small` または `none` のとき
- **THEN** 落ち着いた配色と通常サイズで控えめに表示する。

### Requirement: 目標への前進（アイテム入手による残りの減少）
システムは、比較スナップショット以降の「バッファ込み実効不足の解消ぶん」を推定周回数（前進周回）に換算し、「目標への前進 +N周ぶん」として**周回数を主役**に**プラス表記**で表示しなければならない (SHALL)。内訳として AP相当（素材ごとの最安AP単価＝選択クエスト内の最小消費APをドロップ率で割った値による独立換算の合計）と費用（AP相当を聖晶石168個1万円パック基準で換算した概算金額）を小さく併記する。

算出は**消費中立**でなければならない (SHALL): アイテムごとに現在所持を過去所持で下限クランプしてから不足の減少を数え、育成等による所持の純減は本指標を減少させない。これにより前進周回・AP相当は常に 0 以上となる。素材ごとの独立換算（単価×解消数の合算）であり、ソルバーの再実行を必要としない (SHALL NOT)。本指標は tier 判定の主指標と同一である。

#### Scenario: 前進の算出
- **WHEN** 現在の育成目標の必要数・ストック設定（バッファ量・有効/無効）・現在所持・比較スナップショットの過去所持・ドロップ率が取得できるとき
- **THEN** 素材ごとに実効目標（育成目標の必要数にストックバッファが有効なら加算した値）を求め、現在所持を過去所持で下限クランプした値との差分から実効不足の減少分を算出し、単価を掛けて合算することで前進周回と AP相当を算出する。

#### Scenario: 周回主役での表示と内訳
- **WHEN** 前進周回 > 0 のとき
- **THEN** 「目標への前進 +N周ぶん」を主役として表示し、その下に AP相当と費用を小さく内訳表示する。

#### Scenario: 育成消費が前進を目減りさせないこと
- **WHEN** 比較スナップショット以降に素材を獲得しつつ、育成で一部素材の所持が純減したとき
- **THEN** 純減した素材は過去所持と同値として扱われ、獲得（純増）した素材の分だけが前進に計上される。前進が負になってはならない。

#### Scenario: 過去所持が欠損している場合のフォールバック
- **WHEN** 比較スナップショットに所持データが記録されていないとき
- **THEN** 前進（周回/AP/費用）は算出せず非表示とし、tier は労力周回等で判定する。

### Requirement: 計算根拠の補足表示
システムは、「目標への前進」の計算根拠（素材ごとの単価＝選択クエスト内最高ドロップ率を基準とした換算であること、選択クエストにドロップが無い素材は全クエスト中の最高率で換算すること、入手経路は問わないこと）を、情報アイコン（i）のツールチップ（オーバーレイ）で確認できるようにしなければならない (SHALL)。常時表示の注釈ではなく、アイコン操作時に表示する。

#### Scenario: 計算根拠のツールチップ表示
- **WHEN** 「目標への前進」見出しの情報アイコン（i）にカーソルを合わせる／操作したとき
- **THEN** 素材ごとの単価換算の基準（選択クエスト内最高ドロップ率、フォールバック時は全クエスト中最高率）と入手経路を問わないことを説明するオーバーレイが表示される。

### Requirement: 30/60/90日候補のうちperDay最大の窓を基準とした比較
システムは、期間タブを設けず、比較を「**今（ライブ状態）**」と「**過去スナップショットの複数候補**」の2点で行わなければならない (SHALL)。候補は「30日前」「60日前」「90日前」それぞれに時刻差が最小のスナップショットとし（同一スナップショットに複数候補が解決した場合は1つに畳む）、サーヴァント育成データも所持データも持たない degenerate スナップショットは候補から除外しなければならない (SHALL NOT baseline として採用)。

採用する baseline は、各候補について前進周回を周/日に正規化した値（forwardPerDay）のうち最大のものとする (SHALL)。forwardPerDay が正となる候補が1つも無い場合は、各候補の労力周回を周/日に正規化した値（effortPerDay）のうち最大のものを採用する (SHALL)。前進周回と労力周回の指標は跨いで比較してはならない (SHALL NOT)（forwardPerDay が正の候補が1つでもあれば、effortPerDay がより大きい他候補があっても採用しない）。複数候補の値が同値の場合は、より短い窓（30日 > 60日 > 90日の順）を優先する (SHALL)。

比較基準の時点は見出しの上に経過日数（`昨日` / `N日前`）で明示しなければならない (SHALL)。

#### Scenario: 直近30日にバースト獲得があれば30日窓が採用される
- **WHEN** 30日窓のforwardPerDayが60日窓・90日窓のforwardPerDayより大きいとき
- **THEN** 30日窓の baseline・tier・表示値を採用する。

#### Scenario: 直近30日に入手経路のバーストが乗らなかった場合は長い窓が採用される
- **WHEN** デイリーミッションチケットの一括交換等で獲得がまとまっており、直近30日の窓では未消化のためforwardPerDayが低いが、60日窓または90日窓で見るとforwardPerDayがより高いとき
- **THEN** forwardPerDayが最大となる窓（60日窓または90日窓）の baseline・tier・表示値を採用する。

#### Scenario: 全窓で前進ゼロなら労力周回で比較する
- **WHEN** 30/60/90日いずれの候補もforwardLapsが0以下のとき
- **THEN** 各候補のeffortPerDayのうち最大の候補を採用し、tierを補完する（legendaryには到達させない）。

#### Scenario: 直近のみ存在する場合は最古を候補とする
- **WHEN** スナップショットが全て30日前より新しいとき（例: 今日が 06-07 で 06-02 / 06-03 / 06-06）
- **THEN** 30/60/90日いずれの候補も最も古い 06-02 に解決し、1候補として扱う。

#### Scenario: 中身の無いスナップショットを基準にしない
- **WHEN** 候補がサーヴァント育成データも所持データも持たない degenerate レコードのとき
- **THEN** それを候補から除外し、比較に使える中身を持つ他の候補を基準とする。比較できるものが無ければ初回登録等のフォールバックを表示する。

#### Scenario: 変化ゼロでも実比較は成立する
- **WHEN** 採用された baseline と今で数値変化が無いとき
- **THEN** 「スナップショット無し」ではなく「変化ゼロ（登録お疲れさま）」として扱う。

### Requirement: 結果ページの計算履歴グラフ表示
システムは、ソルバー結果ページ（`/farming/results/[id]`）において、進捗報告パネルの代わりに計算履歴の消費AP推移グラフを表示しなければならない (SHALL)。

#### Scenario: 履歴グラフの表示
- **WHEN** ログイン済みで計算履歴が2件以上あるとき
- **THEN** 結果ページに計算履歴の消費AP推移グラフ（`/api/farming/history` 由来）を表示する。
- **THEN** モード別スタッツ（周回数・消費AP・費用）はそのまま表示する。

#### Scenario: 履歴不足・未ログイン時
- **WHEN** 未ログイン、または計算履歴が2件未満のとき
- **THEN** 履歴グラフは表示しない（自動的に非表示）。

### Requirement: 新規入手サーヴァントの検出と表示
システムは、比較スナップショット間で新規に「所持」化されたサーヴァントを検出し、新規入手を歓迎するメッセージの選択に用いなければならない (SHALL)。検出は、比較対象スナップショットにサーヴァント育成データが記録されている場合にのみ行い、記録が無い場合は新規 0 件として扱わなければならない (SHALL)。

システムは、検出した新規入手サーヴァントを、進捗報告画面において名前付きで一覧表示しなければならない (SHALL)。名前が解決できないサーヴァントは ID で表示し、表示件数が多い場合は上位数件＋残数（「ほか N 騎」）に省略してよい (MAY)。サーヴァントの数量を表す単位は「騎」を用いなければならない (SHALL)。

#### Scenario: 新規入手サーヴァントの一覧表示
- **WHEN** 比較スナップショット間で新規入手サーヴァントが 1 騎以上検出されたとき
- **THEN** 「新しい仲間（N 騎）」として該当サーヴァントの顔アイコン（Atlas の Faces）と名前を一覧表示する（名前未解決時は ID 表示にフォールバックする）。

#### Scenario: ガチャ後の進捗評価
- **WHEN** 比較スナップショット間でサーヴァントの有効化状態が「未所持」から「所持済み」に変化したサーヴァントが存在するとき
- **THEN** 新規入手サーヴァントとして検出し、歓迎メッセージの選択に用いる。
- **THEN** 目標を現在に固定して算出するため、新規サーヴァントの必要素材は現在目標に含まれ、アイテム入手による減少指標を歪めない。

#### Scenario: 比較対象にサーヴァント育成データが無い場合
- **WHEN** 比較対象スナップショットにサーヴァント育成データが存在しないとき
- **THEN** 新規サーヴァントを 0 件として扱う。
- **THEN** 全所持サーヴァントを新規と誤判定しない。

## Constraints
- **アニメーション**: v1 では実装しない。
- **比較基準**: タブは設けず、baseline は「約1ヶ月前に最も近いスナップショット1つ」。前進周回・労力周回はクライアントで素材ごとの単価換算により算出し（ソルバー再実行不要）、スキル合計の増分等はサーバが付与する。
- **マシュ画像**: Atlas Academy CDN から取得する。
