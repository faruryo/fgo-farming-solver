# 仕様書: 進捗ビジュアライザー (Progress Visualizer)

## Purpose
ユーザーのファーミング進捗を可視化し、過去のスナップショットとの比較を通じて達成感をフィードバックする機能。ソルバー実行成功後にモーダルとして表示される。
## Requirements
### Requirement: 進捗報告画面の表示
システムは、ユーザーの育成進捗を報告するマシュ進捗パネルを、ダッシュボードの「あなたの育成進捗」セクション（達成率円グラフの上）に表示しなければならない (SHALL)。比較対象が無い場合も破綻せず案内を表示しなければならない (SHALL)。

#### Scenario: ダッシュボードでの進捗表示
- **WHEN** ダッシュボードを開き、現在の育成状態と進捗データが取得できたとき
- **THEN** 「あなたの育成進捗」セクション上部にマシュ進捗パネルが表示され、下部に既存の達成率円グラフが表示される。

#### Scenario: 比較対象なし時の表示
- **WHEN** 過去スナップショットが存在しないか、指定期間のスナップショットが取得できないとき
- **THEN** 「登録お疲れ様」モードのメッセージとともにパネルが表示される。

### Requirement: 進捗 tier の判定
システムは、進捗の達成度を `large` / `medium` / `small` / `none` の 4 段階（tier）で判定しなければならない (SHALL)。主指標は「目標達成に必要な残りAPの減少量（`reducedAp`）」を経過時間の自然回復AP（5分=1AP）で正規化した値とする。`reducedAp` が算出できない、または 0 以下の期間（育成で素材消費が上回った日・ソルバー未実行など）は、素材スループット（獲得＋育成投入の個数）を経過日数でならした値で tier を補完し、活動のあった日が不当に `none` にならないようにしなければならない (SHALL)。

#### Scenario: reducedAp による判定
- **WHEN** `reducedAp > 0` のとき
- **THEN** `reducedAp >= naturalRecoveryAp * 1.5` → `large`、`naturalRecoveryAp <= reducedAp < naturalRecoveryAp * 1.5` → `medium`、`0 < reducedAp < naturalRecoveryAp` → `small` と判定する。

#### Scenario: スループットによる補完
- **WHEN** `reducedAp` が算出できない、または 0 以下のとき
- **THEN** 素材スループット（獲得＋育成投入の個数）を経過日数でならして tier を判定し、活動があれば `none` にしない。

#### Scenario: 進捗ゼロの判定
- **WHEN** reducedAp・素材スループット・育成総量・新規入手のいずれも無いとき
- **THEN** tier は `none`、フォールバックは `zero_progress` と判定する。

### Requirement: rarity 別の必要AP推定値の事前計算
システムは、各 rarity（1〜5）から代表サーヴァントを抽出して solver で必要APを計算し、その平均値（または中央値）を保持しなければならない (SHALL)。

#### Scenario: rarity-AP テーブルの生成
- **WHEN** 事前計算プロセスが実行されたとき
- **THEN** 各 rarity からランダムに複数体のサーヴァントを選択し、共通目標（再臨/スキル/アペンド）達成までの必要APを solver で算出する。
- **THEN** 結果の平均値または中央値を rarity-AP テーブルとして保持する。

#### Scenario: 高難易度アクセス考慮の2セット計算
- **WHEN** rarity-AP テーブルを生成するとき
- **THEN** 「高難易度なし」と「高難易度あり（オーディールコール、冠位戦含む）」の 2 セットを生成する。

### Requirement: 高難易度コンテンツアクセス可否の判定
システムは、ユーザーの選択クエストリスト（`checkedQuests`）から高難易度コンテンツ（オーディールコール、冠位戦）への到達可否を推測しなければならない (SHALL)。

#### Scenario: アクセス可否の推測
- **WHEN** 進捗計算を行うために rarity-AP テーブルを選択する必要があるとき
- **THEN** ユーザーの `checkedQuests` に該当コンテンツの代表クエストIDが含まれるかをチェックする。
- **THEN** 含まれる場合は「高難易度あり」テーブル、含まれない場合は「基本」テーブルを使用する。

### Requirement: この期間の活動の表示（獲得素材・スキル合計・育成総量）
システムは、比較スナップショット以降の育成活動を「この期間の活動」として表示しなければならない (SHALL)。指標は (a) 獲得した素材の合計個数（`itemsFarmed`、QP除外）、(b) 所持サーヴァントのスキル現在レベル合計の増分（`skillDelta`、新規入手サーヴァントを含む）、(c) 育成目標レンジ（再臨/スキル/アペンド）が縮んだ合計（育成総量）であり、いずれも値が正のものだけ表示する。

#### Scenario: スキル合計の変化の表示
- **WHEN** 所持サーヴァント（新規入手を含む）のスキル現在レベル合計が比較スナップショットから増えたとき
- **THEN** その増分を「スキル合計 +N」として表示する。比較対象に存在しない新規サーヴァントのスキルも増分として数える。

#### Scenario: 育成総量の表示
- **WHEN** 比較スナップショット間でサーヴァントの育成目標レンジ合計が縮んだとき
- **THEN** 縮んだ量の合計を「育成総量」として表示する。

### Requirement: 素材スループットの算出
システムは、比較スナップショット以降の所持数の増減から素材スループットを算出しなければならない (SHALL)。所持が増えた分の合計を「獲得素材（`itemsFarmed`）」、減った分の合計を「育成投入（`itemsConsumed`）」とし、いずれも QP（atlasId `1`）は所持規模が他素材を桁違いに上回るため除外する。スループットは tier の補完判定（reducedAp が無い/0以下のとき）に用いる。

#### Scenario: QP の除外
- **WHEN** 所持差分に QP（atlasId `1`）の増減が含まれるとき
- **THEN** QP の増減はスループットに含めない。

### Requirement: マシュによる進捗フィードバック
進捗報告画面において、マシュ・キリエライトの画像と進捗種別に応じたメッセージを表示しなければならない (SHALL)。

#### Scenario: tier 別マシュメッセージ
- **WHEN** 進捗 tier が `large` / `medium` / `small` / `none` のいずれかに判定されたとき
- **THEN** その tier に対応する事前定義済みマシュメッセージ群からランダムにメッセージが選択され、マシュの画像とともに表示される。

#### Scenario: 「お疲れ様」モード
- **WHEN** 過去スナップショットがない、または全指標で改善ゼロのとき
- **THEN** マシュの「登録お疲れ様」系メッセージが表示される。

#### Scenario: 新規入手・育成は tier に依らず労う
- **WHEN** 新規入手サーヴァントが居る、または育成総量が正のとき
- **THEN** tier が `none` であっても「休んでください」系ではなく、新規入手の歓迎／育成を労うメッセージを優先して表示する。

### Requirement: 達成感の視覚的演出
進捗 tier に応じて、モーダル内の色・サイズ・装飾の強度を変化させなければならない (SHALL)。アニメーションは v1 では実装しない。

#### Scenario: tier 別ビジュアル
- **WHEN** tier が `large` のとき
- **THEN** 派手な配色と大きい数値表示で達成感を強調する。

#### Scenario: 控えめな表示
- **WHEN** tier が `small` または `none` のとき
- **THEN** 落ち着いた配色と通常サイズで控えめに表示する。

### Requirement: 目標への前進（アイテム入手による残りの減少）
システムは、比較スナップショット以降にアイテムを入手したことで「目標達成に必要な残りAP・周回数・費用」がどれだけ減ったかを、目標を現在に固定した再ソルブで算出し、「目標への前進」として**プラス表記**で表示しなければならない (SHALL)。AP（`reducedAp`）を主役の大きな数値とし、周回数・費用はその単位換算として小さく内訳表示する。AP・費用は消費AP最小プラン、周回数は周回数最小プランで算出するため両者の増減はずれることがあり、周回数が 0 以下（減っていない）の場合は表示してはならない (SHALL NOT)。目標を両辺とも現在値に固定するため、目標を増やしても本指標は増加してはならない (SHALL NOT)。本指標 `reducedAp` は tier 判定の主指標でもある。

#### Scenario: 減少量の算出
- **WHEN** 現在の目標（`material/result`）と、現在の所持（`posession`）および比較スナップショットの過去所持が取得できるとき
- **THEN** `need = max(0, 現在目標 − 所持)` を所持別に作り、現在所持版と過去所持版をそれぞれソルバーで解く。
- **THEN** `reducedAp = solve(現在目標 − 過去所持).total_ap − solve(現在目標 − 現在所持).total_ap`（消費AP最小）、減少周回 `= 同 total_lap`（周回数最小）、減少費用 `= round(reducedAp / 144 / 168 * 10000)` を算出する。

#### Scenario: プラス表記での表示と内訳
- **WHEN** `reducedAp > 0` のとき
- **THEN** 「目標への前進 +N AP相当」を主役として表示し、その下に周回数（>0 のときのみ「周回 N周」）と費用（「費用 ¥N」）を小さく内訳表示する。

#### Scenario: 目標増加が減少量に影響しないこと
- **WHEN** 比較スナップショット以降に育成目標が増えた（`material/result` の総数が増加した）とき
- **THEN** 目標は両辺とも現在値で固定されているため、目標増加そのものは reducedAp・減少周回を増減させない。

#### Scenario: 過去所持が欠損している場合のフォールバック
- **WHEN** 比較スナップショットに `posession` が記録されていないとき
- **THEN** 目標への前進（AP/周回/費用）は算出せず非表示とし、tier はスループット等で判定する。

### Requirement: 計算根拠の補足表示
システムは、「目標への前進」の計算根拠（AP・費用は消費AP最小、周回数は周回数最小の別プランで算出するため結果がずれ得ること、周回数が減っていない場合は非表示とすること）を、情報アイコン（i）のツールチップ（オーバーレイ）で確認できるようにしなければならない (SHALL)。常時表示の注釈ではなく、アイコン操作時に表示する。

#### Scenario: 計算根拠のツールチップ表示
- **WHEN** 「目標への前進」見出しの情報アイコン（i）にカーソルを合わせる／操作したとき
- **THEN** AP・費用と周回数が別プラン算出でずれ得ること、周回数が減っていない場合は非表示にすることを説明するオーバーレイが表示される。

### Requirement: 約1ヶ月前に最も近いスナップショットを基準とした単一比較
システムは、期間タブを設けず、比較を「**今（ライブ状態）**」と「**約1ヶ月前（30日前）に最も近い過去スナップショット1つ**」の2点で行わなければならない (SHALL)。ちょうど30日前のスナップショットは通常存在しないため、30日前との時刻差が最小のものを baseline として選ぶ。手持ちが全て直近（1ヶ月未満）の場合は結果として最も古いものが選ばれ、可能な限り長い期間の積み上げを表示する。比較基準の時点は見出しの上に経過日数（`昨日` / `N日前`）で明示しなければならない (SHALL)。

`material` も `posession` も持たない degenerate スナップショット（旧経路が書いた `{items,quests}` のみのレコード等）は、いかなる進捗指標も算出できないため baseline として採用してはならない (SHALL NOT)。

#### Scenario: 直近のみ存在する場合は最古を基準にする
- **WHEN** スナップショットが全て30日前より新しいとき（例: 今日が 06-07 で 06-02 / 06-03 / 06-06）
- **THEN** 30日前に最も近い＝最も古い 06-02 を baseline として選び、「5日前と比べて」と表示する。

#### Scenario: 30日前をまたぐ場合は30日前に最も近いものを選ぶ
- **WHEN** 30日前付近の前後にスナップショットが存在するとき
- **THEN** 30日前との時刻差が最小のスナップショットを baseline として選ぶ。

#### Scenario: 中身の無いスナップショットを基準にしない
- **WHEN** baseline 候補が `material` も `posession` も持たない degenerate レコードのとき
- **THEN** それを採用せず、比較に使える中身を持つスナップショットを基準とする。比較できるものが無ければ初回登録等のフォールバックを表示する。

#### Scenario: 変化ゼロでも実比較は成立する
- **WHEN** baseline と今で数値変化が無い（`zero_progress`）とき
- **THEN** 「スナップショット無し」ではなく「変化ゼロ（登録お疲れさま）」として扱う。

### Requirement: 結果ページの計算履歴グラフ表示
システムは、ソルバー結果ページ（`/farming/results/[id]`）において、進捗報告パネルの代わりに計算履歴の消費AP推移グラフを表示しなければならない (SHALL)。

#### Scenario: 履歴グラフの表示
- **WHEN** ログイン済みで計算履歴が2件以上あるとき
- **THEN** 結果ページに計算履歴の消費AP推移グラフ（`/api/farming/history` 由来）を表示する。
- **THEN** モード別スタッツ（周回数・消費AP・費用）はそのまま表示する。

#### Scenario: 履歴不足・未ログイン時
- **WHEN** 未ログイン、または計算履歴が2件未満のとき
- **THEN** 履歴グラフは表示しない（自動的に非表示）。

### Requirement: 新規入手サーヴァントの検出と表示
システムは、比較スナップショット間で新規に「所持」化されたサーヴァントを検出し、新規入手を歓迎するメッセージの選択に用いなければならない (SHALL)。検出は、比較対象スナップショットに chaldea state（material）が記録されている場合にのみ行い、記録が無い場合は新規 0 件として扱わなければならない (SHALL)。

システムは、検出した新規入手サーヴァントを、進捗報告画面において名前付きで一覧表示しなければならない (SHALL)。名前が解決できないサーヴァントは ID で表示し、表示件数が多い場合は上位数件＋残数（「ほか N 騎」）に省略してよい (MAY)。サーヴァントの数量を表す単位は「騎」を用いなければならない (SHALL)。

#### Scenario: 新規入手サーヴァントの一覧表示
- **WHEN** 比較スナップショット間で新規入手サーヴァントが 1 騎以上検出されたとき
- **THEN** 「新しい仲間（N 騎）」として該当サーヴァントの顔アイコン（Atlas の Faces）と名前を一覧表示する（名前未解決時は ID 表示にフォールバックする）。

#### Scenario: ガチャ後の進捗評価
- **WHEN** 比較スナップショット間で chaldea state の `disabled` が `true` から `false` に変化したサーヴァントが存在するとき
- **THEN** 新規入手サーヴァントとして検出し、歓迎メッセージの選択に用いる。
- **THEN** 目標を現在に固定して算出するため、新規サーヴァントの必要素材は現在目標に含まれ、アイテム入手による減少指標を歪めない。

#### Scenario: 比較対象に chaldea state が無い場合
- **WHEN** 比較対象スナップショットに chaldea state（material）が存在しない（`null`）とき
- **THEN** 新規サーヴァントを 0 件として扱う。
- **THEN** 全所持サーヴァントを新規と誤判定しない。

## Constraints
- **アニメーション**: v1 では実装しない。
- **比較基準**: タブは設けず、baseline は「約1ヶ月前に最も近いスナップショット1つ」。`reducedAp`（目標固定再ソルブ）はクライアントで算出し、`skillDelta` 等はサーバが付与する。
- **マシュ画像**: Atlas Academy CDN から取得する。
