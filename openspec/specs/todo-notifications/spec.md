# 仕様書: TODO通知 (TODO Notifications)

## Purpose
デイリー/ウィークリーミッションやイベント交換の期限切れによる取りこぼしを防ぐため、TODO の自動生成・手動管理と、期限間近の Web Push 通知を提供する。あわせて Web Push の環境ゲーティング（対応可否の検出）とユーザーガイダンスの要件を定義する。

## Requirements

### Requirement: 自動 TODO 追加 (Auto TODO Generation)
システムは、ユーザー向けに以下の標準TODOタスクを自動的に生成しなければならない (SHALL)。
1. デイリーミッション: 毎日午後11:59 JSTに期限切れ、午前0:00 JSTにリセット。
2. ウィークリーミッション: 毎週日曜日午後11:59 JSTに期限切れ/リセット。
3. イベント交換ショップ: 開催中のイベントの交換期限終了時刻に期限切れ。

自動生成されるタスクは、該当する期間の開始時に初期化され、未完了（未チェック）の状態で作成される。

#### Scenario: デイリーミッションの自動生成
- **WHEN** ユーザーがダッシュボードまたはTODO管理画面を表示し、前回のログイン日と異なる新しいJST日付（午前0:00基準）が開始しているとき
- **THEN** 「デイリーミッションをクリアする」というタスクが自動生成される。

#### Scenario: ウィークリーミッションの自動生成
- **WHEN** ユーザーがダッシュボードまたはTODO管理画面を表示し、新しい週（月曜日午前0:00 JST基準）が開始しているとき
- **THEN** 「ウィークリーミッションをクリアする」というタスクが自動生成される。

#### Scenario: イベント交換ショップタスクの自動生成
- **WHEN** 開催中のイベントが存在し、そのイベントのアイテム交換ショップの終了期限（`DashboardEvent.shopFinishedAt`。現状は `endedAt` のエイリアス、design.md Non-Goals 参照）が設定されているとき
- **THEN** 「[イベント名] アイテム交換を完了する」というタスクが自動生成され、期限にイベント交換終了日時が設定される。

### Requirement: 自動追加の ON/OFF 設定 (Auto-Add Settings)
システムは、ユーザー設定画面において、自動 TODO 生成の有効/無効をカテゴリ（デイリー、ウィークリー、イベント交換）ごとに個別にトグルで切り替える設定項目を提供しなければならない (SHALL)。プッシュ通知については、全カテゴリ一括の有効/無効トグルを1つ提供する。このプッシュ通知設定は、クラウド同期の対象から除外し、専用の端末ローカルキー（`fgo_push_enabled`）でのみ保持されなければならない (SHALL)。

#### Scenario: 自動生成の無効化
- **WHEN** ユーザーが設定画面で「デイリーミッション自動追加」を無効（OFF）に設定したとき
- **THEN** 以降のデイリーミッション TODO は自動生成されない。また、現在未完了として存在する自動生成済みのデイリーミッション TODO は非表示または削除される。

#### Scenario: 設定の同期
- **WHEN** ユーザーが設定を変更したとき
- **THEN** デイリーミッション、ウィークリーミッション、およびイベント交換の自動追加設定値は localStorage の `todoSettings` キーに保存され、既存のオートシンク機構を通じて Cloudflare KV に同期される。ただし、プッシュ通知の ON/OFF トグル状態は `todoSettings` および同期データに含めず、端末ローカルキー `fgo_push_enabled` にのみ保持される。

### Requirement: プッシュ通知の許諾と登録 (Push Notification Permission and Subscription)
システムは、ブラウザの Web Push API を使用して、ユーザーのブラウザから通知許可を取得し、プッシュ通知の購読情報をサーバーに登録しなければならない (SHALL)。

#### Scenario: プッシュ通知の有効化
- **WHEN** ユーザーが設定画面で「プッシュ通知を有効にする」ボタンをクリックしたとき
- **THEN** ブラウザが通知許可を求め、許可された場合は Service Worker 経由で PushSubscription を作成する。
- **THEN** 作成された購読情報（Endpoint, Keys）をサーバーの API `/api/notifications/subscribe` へ送信し、D1 データベースに保存する。

#### Scenario: 購読情報の失効処理
- **WHEN** 配信スクリプト（GitHub Actions 上の通知ディスパッチャ）がプッシュ通知を送信した際に、ブラウザ側の購読が失効している（410 Gone などのエラーが返る）とき
- **THEN** 配信スクリプトは D1 データベースから該当する PushSubscription レコードおよび対応する `notification_log` レコードを削除する。

### Requirement: 期限間近のプッシュ通知 (Deadline Near Notifications)
システムは、完了していない自動生成 TODO の期限が近づいた場合、Web Push 通知をユーザーに配信しなければならない (SHALL)。
配信可否は、該当ユーザーの購読レコードが D1 に存在するかどうかで判定しなければならず (SHALL)、`todoSettings.pushEnabled` を配信条件として参照してはならない (MUST NOT)。
通知はカテゴリごとの通知開始時刻（期限 − 閾値）から期限までの間に配信バッチの実行が1回でもあれば配信されなければならない (SHALL)。期限を過ぎたタスクへの通知は配信してはならない (MUST NOT)。閾値は以下とする。
- デイリーミッション: 期限切れの2時間前（通知開始 毎日午後9:59 JST。就寝前かつ実締切前に受け取れるようにするための設定）
- ウィークリーミッション: 期限切れの12時間前（通知開始 毎週日曜日午前11:59 JST）
- イベント交換ショップ: 期限切れの24時間前
- カスタムタスク: 期限の24時間前

#### Scenario: デイリーミッション期限間近の通知配信
- **WHEN** 配信バッチ（GitHub Actions ワークフロー等の定期実行）が通知開始時刻（午後9:59 JST）から期限（午後11:59 JST）までの間に実行され、デイリーミッション TODO が未完了かつ、該当ユーザーの購読レコードが D1 に存在するとき
- **THEN** 「デイリーミッションの期限が近づいています！」という通知が配信される。

#### Scenario: ウィークリーミッション期限間近の通知配信
- **WHEN** 配信バッチが通知開始時刻から期限までの間に実行され、ウィークリーミッション TODO が未完了かつ、該当ユーザーの購読レコードが D1 に存在するとき
- **THEN** 「ウィークリーミッションの期限が近づいています！」という通知が配信される。

#### Scenario: イベント交換期限間近の通知配信
- **WHEN** 配信バッチが実行され、イベント交換 TODO の期限が残り24時間未満で未完了であり、かつ該当ユーザーの購読レコードが D1 に存在するとき
- **THEN** 「[イベント名] の交換期限が近づいています！」という通知が配信される。

#### Scenario: バッチ実行が遅延・欠落しても期限前なら配信される
- **WHEN** 通知開始時刻を含む1時間以内に配信バッチの実行が無く（cron の遅延・欠落）、その後期限前に配信バッチが実行されたとき
- **THEN** 未送信の期限間近通知はその実行で配信される（通知開始直後の1時間に実行が無かったことは欠落の理由にならない）。

#### Scenario: 期限超過後は配信しない
- **WHEN** タスクの期限を過ぎてから配信バッチが実行されたとき
- **THEN** そのタスクの期限間近通知は配信されない。

#### Scenario: アプリ未起動ユーザーへの通知
- **WHEN** ユーザーが当日アプリを一度も開いておらず、自動生成タスクが `todoState` に未生成だが、配信スクリプトが `lib/todo/period.ts` を用いて当該期間の期待タスク（デイリー/ウィークリー/イベント/カスタム）を独立に再計算した結果、対応する自動生成カテゴリ（`autoDaily`/`autoWeekly`/`autoEvent`）が有効かつ、該当ユーザーの購読レコードが D1 に存在するとき
- **THEN** そのタスクは未完了とみなされ、期限間近の通知が配信される（クライアント側でタスクが未生成であることは通知をスキップする理由にならない）。

#### Scenario: 通知の重複防止
- **WHEN** 配信バッチが同一の `notification_key`（タスクID）に対して複数回実行されるとき（拡張された通知窓内で複数回実行される場合を含む）
- **THEN** 通知は `notification_log` への `INSERT ... ON CONFLICT DO NOTHING` によって最初の1回のみ配信され、以降の実行では新規挿入されないためスキップされる。

### Requirement: 手動チェックと永続化 (Manual Task Check-off and Sync)
ユーザーは、自動生成された TODO および手動で追加した TODO の完了状態（チェックボックス）を切り替えることができ、その状態はクラウド間で同期されなければならない (SHALL)。

#### Scenario: タスクの完了チェック
- **WHEN** ユーザーが TODO アイテムのチェックボックスをオン（完了）にしたとき
- **THEN** 該当タスクのステータスが「完了」に変更され、localStorage が更新され、オートシンク経由で `/api/cloud` に送信される。
- **THEN** 期限間近になってもこのタスクに関するプッシュ通知は送信されない。

### Requirement: カスタムタスクの管理 (Custom Task Management)
ユーザーは、任意のタイトルと期限日時を持つカスタム TODO タスクを作成・編集・削除できなければならない (SHALL)。カスタムタスクは `category: 'custom'` として `todoState` に保存され、既存のオートシンク機構を通じて Cloudflare KV に同期される。

#### Scenario: カスタムタスクの作成
- **WHEN** ユーザーが TODO 管理画面でタイトルと期限日時を入力してカスタムタスクを追加したとき
- **THEN** UUID を ID とする未完了のカスタムタスクが生成され、localStorage の `todoState` に保存されオートシンクされる。

#### Scenario: カスタムタスクの編集・削除
- **WHEN** ユーザーがカスタムタスクのタイトルまたは期限を変更、あるいは削除したとき
- **THEN** 変更内容が `todoState` に反映され、オートシンク経由で `/api/cloud` に送信される。

#### Scenario: カスタムタスクの期限通知
- **WHEN** 配信バッチが実行され、未完了のカスタムタスクの期限まで24時間を切っており（期限前）、かつ該当ユーザーの購読レコードが D1 に存在するとき
- **THEN** 「[タスク名] の期限が近づいています！」という通知が、当該タスクにつき1回だけ配信される（`notification_log` により重複配信を防止する）。

### Requirement: 未ログインユーザーのプッシュ通知ゲーティング (Guest Push Gating)
プッシュ通知の購読はサーバー側 `user_id`（Google ログインの ID）に紐付くため、システムは未ログイン状態でのプッシュ通知有効化を防がなければならない (SHALL)。

#### Scenario: 未ログイン時のプッシュ通知トグル無効化
- **WHEN** ユーザーがログインしていない状態で通知設定画面を開いたとき
- **THEN** 「プッシュ通知を有効にする」トグル/ボタンは無効（disabled）で表示され、「プッシュ通知にはログインが必要です」という案内と Google サインイン導線（`signIn('google')`、`components/common/auth-button.tsx` と同様のパターン）が表示される。

#### Scenario: ログイン後の有効化
- **WHEN** ユーザーがサインインを完了したとき
- **THEN** プッシュ通知トグルが有効化され、通常の許諾・購読フローが利用可能になる。

### Requirement: 非対応環境でのプッシュ通知ゲーティング (Unsupported Environment Push Gating)
システムは、プッシュ通知トグルの操作を受け付ける前に Web Push の利用可否（Service Worker / PushManager / Notification API の存在）をクライアント側で検出しなければならない（MUST）。非対応環境ではトグルを無効化し、例外による失敗を発生させてはならない（MUST NOT）。

#### Scenario: 非対応ブラウザでのトグル無効化
- **WHEN** Web Push API が利用できないブラウザで TODO 通知設定を表示する
- **THEN** プッシュ通知トグルは disabled 表示になり、操作しても登録処理は実行されない

#### Scenario: 対応環境での従来動作の維持
- **WHEN** Web Push API が利用できるブラウザ（デスクトップ Chrome、ホーム画面追加済み iOS PWA 等）で設定を表示する
- **THEN** トグルは従来どおり操作でき、許諾 → 購読 → サーバー登録のフローが実行される

### Requirement: iOS 向けホーム画面追加ガイダンス (iOS Add-to-Home-Screen Guidance)
システムは、Web Push 非対応かつ iOS 系デバイス（iPhone / iPad）と判定した場合、共有メニューから「ホーム画面に追加」し追加したアイコンから開くことでプッシュ通知を利用できる旨の案内を通知設定内に表示しなければならない（MUST）。iOS 系以外の非対応環境には、ブラウザが対応していない旨の汎用メッセージを表示しなければならない（MUST）。

#### Scenario: iPhone の Safari タブでの案内表示
- **WHEN** iPhone の Safari（ホーム画面未追加）で TODO 通知設定を表示する
- **THEN** 無効化されたトグルとともに「ホーム画面に追加」の手順案内が表示される

#### Scenario: iOS 以外の非対応ブラウザでの汎用メッセージ
- **WHEN** iOS 系以外で Web Push 非対応のブラウザで TODO 通知設定を表示する
- **THEN** 「このブラウザはプッシュ通知に対応していません」相当の汎用メッセージが表示される

#### Scenario: ホーム画面追加済み iOS PWA では案内を出さない
- **WHEN** ホーム画面に追加した PWA として iOS で開き、TODO 通知設定を表示する
- **THEN** 案内文は表示されず、トグルが通常どおり操作できる

### Requirement: プッシュ登録失敗の原因別フィードバック (Reason-Specific Push Failure Feedback)
システムは、プッシュ通知の登録が失敗した場合、失敗理由（通知許可の拒否 / 購読の失敗 / サーバーへの登録失敗）を区別したエラーメッセージを表示しなければならない（MUST）。メッセージには、ユーザーが問題解決のために次に取るべき具体的なアクション（iOSでのPWA再インストール案内、PC等のブラウザ設定変更方法、通信エラー・セッション切れ時のログイン再試行など）を含めなければならない（MUST）。

#### Scenario: 通知許可を拒否した場合
- **WHEN** トグル ON 後の許諾ダイアログでユーザーが通知を拒否する
- **THEN** 通知が許可されていない旨のエラーメッセージが表示され、iOSではホーム画面アプリの再インストールが必要であること、iOS以外ではブラウザ設定でブロックを解除する手順が必要であることの具体的な案内が表示され、トグルは OFF のまま維持される。

#### Scenario: サーバー登録または購読に失敗した場合
- **WHEN** 購読情報の生成やサーバー登録（API送信）がエラーになる
- **THEN** 通信環境や一時的なセッション切れの可能性がある旨のエラーメッセージが表示され、再試行や再ログインを促す案内が表示され、トグルは OFF のまま維持される。

#### Scenario: 購読解除 API に失敗した場合
- **WHEN** ユーザーがトグルを OFF にし、ブラウザ側の購読解除は完了したが D1 への DELETE が失敗する
- **THEN** 解除に失敗した旨と再試行を促す案内が表示され、次回の有効化操作では既存のブラウザ購読を再利用して D1 へ再登録できる。

#### Scenario: エラーメッセージの i18n
- **WHEN** 言語設定を英語にして上記いずれかの失敗が発生する
- **THEN** 対応する英語のメッセージが表示される
