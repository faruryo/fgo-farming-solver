## Requirements
### Requirement: イベントデータの取り込み
システムは、開催中（および直近終了）のロト型イベントについて、Atlas Academy の `nice_event` からロトボックス・交換所・イベント通貨の情報を取得し、コンパクト化して KV (`event_data_json`) に保存しなければならない (SHALL)。取得・整形は GitHub Actions 定期ワークフローで行い、Cloudflare Workers では KV を読むだけとする。

#### Scenario: ロトボックスと箱コストの取得
- **WHEN** 取り込みワークフローが対象イベントを処理するとき
- **THEN** 各ロトの「1箱あたりの確定内容（アイテムと個数）」「1箱あたりのコスト（イベント通貨と個数）」「箱の最大数（`maxNum`）」「レアボックス判定（`isRare`）」が KV に保存される。

#### Scenario: 交換所の取得
- **WHEN** 取り込みワークフローが対象イベントを処理するとき
- **THEN** 交換所アイテムの「獲得物」「コスト（通貨・個数）」「在庫上限（`limitNum`）」が KV に保存される。

#### Scenario: スキーマ不整合のスキップ
- **WHEN** あるイベントの `nice_event` がバリデーションに失敗したとき
- **THEN** そのイベントはスキップされ、他のイベントの取り込みと既存 KV データは影響を受けない。

#### Scenario: 既存恒常データの不可侵
- **WHEN** イベントデータを取り込むとき
- **THEN** 恒常ドロップデータ (`all_drops_json`) と `getWars` の `eventId == 0` フィルタは変更されない。

### Requirement: 既存ロスターの不足素材からの箱数目標設定
システムは、ボックスイベントについて、**既存の永続ロスター**（`ChaldeaState` ＋所持数）から算出される不足素材に対し、必要箱数とその育成インパクト（確定報酬・交換所が不足素材をどれだけ充当するか）を提示しなければならない (SHALL)。本機能は永続ロスターを**読み取り専用**で利用し、ロスター・所持データを書き換えてはならない (SHALL NOT)。

#### Scenario: 既存ロスターの不足素材からの箱数提示
- **WHEN** ログイン済み／ロスター設定済みのユーザーがボックスイベントの計画ページを開いたとき
- **THEN** 既存 `ChaldeaState` ＋所持数から `get-materials`/`diff-materials` で不足素材を求め、ボックス確定報酬・交換所がそれをどれだけ充当するか（育成インパクト）と、満たすために必要な最小箱数・通貨・周回数・消費APが提示される。

#### Scenario: ロスターへの非書き込み
- **WHEN** イベント計画を算出・操作するとき
- **THEN** `ChaldeaState`・所持数・クラウド同期データは一切書き換えられない（read-only）。

#### Scenario: ロスター未設定時のフォールバック
- **WHEN** 未ログインまたはロスター未設定で計画ページを開いたとき
- **THEN** 育成インパクト表示は省略され、欲しい素材の直接指定（または目標箱数）による算出にフォールバックする。

### Requirement: ロト計画の算出
システムは、選択されたロト型イベントについて、ユーザー入力（目標箱数 または 欲しいアイテム数、所持イベント通貨、周回ノードと1周あたりのドロップ）から、必要イベント通貨・必要周回数・消費AP・確定報酬を算出しなければならない (SHALL)。

#### Scenario: 目標箱数からの必要通貨算出
- **WHEN** ユーザーが目標箱数 N を入力したとき
- **THEN** 必要イベント通貨 = `N × 1箱あたりコスト − 所持通貨`、確定獲得素材 = `N × 1箱内容` が算出される。

#### Scenario: 欲しいアイテム数からの必要箱数逆算
- **WHEN** ユーザーがロト/交換所で欲しいアイテムの目標個数を入力したとき
- **THEN** その個数を満たす最小の箱数（および交換所配分）と、それに必要なイベント通貨が算出される。
- **WHEN** 交換所アイテムが在庫上限 (`limitNum`) を超える要求のとき
- **THEN** 上限までで打ち切り、不足する旨を警告表示する。

#### Scenario: 周回層は既存ソルバで解く
- **WHEN** 必要イベント通貨（および直接欲しいイベント素材）が確定したとき
- **THEN** イベント通貨/素材を需要、周回ノードを変数とした `Drops` 互換データを合成して既存 `solve()` に渡し、最小AP/最小周回の周回計画を得る。
- **THEN** `solve()` 本体および恒常ソルバ (`/farming`) の挙動は変更されない。

#### Scenario: ドロップデータのフォールバックと明示
- **WHEN** 対象イベント周回ノードの Atlas ドロップデータが存在しないとき
- **THEN** ユーザーが「1周あたりの通貨/素材ドロップ」を手入力でき、その入力に基づいて周回数が算出される。
- **THEN** 算出結果には、使用したドロップ値のソース（Atlas / 手入力）と概算である旨が表示される。

#### Scenario: 確定報酬と周回ドロップの内訳分離
- **WHEN** 計画結果を表示するとき
- **THEN** 「ボックス確定報酬」と「周回ドロップ見込み」は内訳として分離表示され、需要充足判定は両者の合算で行われる。

#### Scenario: レアボックス報酬の扱い
- **WHEN** ボックスにレアボックス（`isRare`）の報酬が含まれるとき
- **THEN** レア報酬が**素材**の場合は確定報酬として充当計算に含める。
- **THEN** レア報酬が**サーヴァント／礼装／コマンドコード**の場合は「N箱で確定入手」として素材充当とは分離して表示する。

#### Scenario: 交換所の在庫上限での打ち切り
- **WHEN** 不足素材を交換所で充当する際に必要数が `limitNum` を超えるとき
- **THEN** 充当は `limitNum` までで打ち切り、超過分は警告のうえ残需要として周回計画（ソルバ）に回される。

### Requirement: ロト計画 UI
システムは、開催中（および直近終了）のロト型イベントの一覧と、個別イベントの計画ページを提供しなければならない (SHALL)。MVP では既存の恒常導線 (`/material` → `/farming`) を変更しない独立ルート (`/events`) として提供する。一覧・計画の各ページは、URL 直打ち以外にアプリ内の常設導線（グローバルナビ等）から到達可能でなければならず (SHALL)、どのページもオーファン（到達不能）であってはならない (SHALL NOT)。

#### Scenario: イベント一覧
- **WHEN** ユーザーが `/events` を開いたとき
- **THEN** 取り込み済みのロト型イベントが会期情報とともに一覧表示され、会期外（終了済み）は明示される。

#### Scenario: 一覧ページへの常設導線（オーファン禁止）
- **WHEN** ユーザーがアプリのどの画面からでもロトイベント機能を探すとき
- **THEN** グローバルナビゲーション（`components/common/nav.tsx`）に `/events` 一覧への常設エントリが存在し、URL 直打ちなしで一覧ページへ到達できる。
- **AND** ダッシュボードからの個別イベント直行導線（`/events/[id]`）は一覧導線を代替せず、両者が併存する。

#### Scenario: 計画ページへの遷移
- **WHEN** ユーザーが一覧からイベントを選ぶ、またはダッシュボードの該当イベントから遷移したとき
- **THEN** そのイベントの計画ページ (`/events/[id]`) が開き、入力フォームと算出結果が表示される。

#### Scenario: 取り込みデータ未存在時の表示
- **WHEN** 対象イベントの KV データが存在しないとき
- **THEN** 計画ページはエラーにならず、データ未取得である旨のプレースホルダを表示する。
