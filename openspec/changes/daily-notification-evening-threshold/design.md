## Context

notification-window-until-deadline で通知窓は [dueAt, deadline) に拡張されたが、デイリーの dueAt = 期限(03:59 JST) − 3h = 00:59 JST のため窓全域が深夜帯にあり、初配信は 2:05 JST だった。advisor レビューでもフォローアップ（閾値を早めて夜帯へ移す）として記録済み。

## Goals / Non-Goals

**Goals:**
- デイリー通知の通知開始を前日 22:59 JST に移し、通常の配信を就寝前の時間帯（23時前後）にする。

**Non-Goals:**
- 配信時刻の厳密な保証（GH Actions cron の遅延特性は変わらない。cron 頻度も予算制約で据え置き）。
- ウィークリー・イベント・カスタムの閾値変更。
- クワイエットアワー（深夜帯の送信抑制）の導入。窓後半が深夜に残るのは仕様とする（未送信のまま朝を迎えるより深夜でも届く方を優先。拡張窓の設計思想を維持）。

## Decisions

### 1. `THRESHOLD_MS.daily` を 3h → 5h に変更
dueAt = 03:59 − 5h = 22:59 JST。実測のバッチ実行間隔（1〜3.5h）では、概ね 23:00〜1:00 頃に配信される。窓は [22:59, 03:59) の5時間となり、最大実行間隔 3.5h を上回るため配信漏れは構造上発生しない。

- 文言は notification-window-until-deadline で非定量化済み（「期限が近づいています」）のため、閾値変更による文言修正は不要。

## Risks / Trade-offs

- **[Risk]** 22時台にデイリーを消化し忘れているだけのユーザーには、従来より約2時間早い「急かし」になる。
- **[Mitigation]** 通知は未完了タスクにのみ飛ぶ。就寝前に受け取れる価値が上回ると判断（ユーザー要望そのもの）。
- **[Risk]** cron 遅延が大きい日は依然として深夜配信になり得る。
- **[Mitigation]** 発生確率は窓の前倒しで大幅に低下する。厳密な時刻保証は Non-Goal。
