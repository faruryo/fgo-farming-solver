## Context

イベント/ロト計画は、構造的に2層の問題に分解できる。

1. **ボックス層（決定論）**: ロトは「1箱 = 固定内容（素材・種火・QP 等の確定報酬）」を、「1箱あたり固定コスト（イベント通貨 N 個）」で開封する。N 箱開けると `N × 箱内容` が確定で手に入り、`N × 箱コスト` の通貨を要する。交換所（shop）も「通貨 → アイテム（在庫上限あり）」の決定論的変換。
2. **周回層（最適化）**: 必要なイベント通貨（＋直接欲しいイベント素材）を、複数の周回ノードから最小AP/最小周回で集める。**これは既存 `lib/solver.ts` の `solve()` と同型**（需要＝アイテム必要数、変数＝ノード周回数、係数＝1周あたりドロップ）。

つまり本機能は「ボックス層で需要（必要通貨）を確定 → 周回層を既存ソルバに委譲」で成立する。新規に解くべき最適化エンジンはなく、リスクの所在は**最適化ではなくデータ取得**にある。

データ取得基盤は既に存在する: 重い Atlas データ（23MB の nice_war）は GitHub Actions で取得・コンパクト化して KV に置き、Workers は KV を読むだけ（`refresh-nice-war.yml` / `lib/master-data/update.ts`）。無料プラン Workers は CPU 10ms 超で確率 kill されるため、この分業は厳守する。

## Goals / Non-Goals

**Goals:**
- ロト型イベントについて、**目標箱数 or 欲しいアイテム数 → 必要イベント通貨・周回数・消費AP・確定報酬**を算出する。
- 既存LPソルバ（`solve()`）を**無改修で再利用**する。
- イベントデータ取り込みを既存の「GH Actions で取得・KV キャッシュ」方式に乗せ、Workers に重い処理を持ち込まない。
- 既存の恒常データ（`all_drops_json`）・恒常ソルバ・素材計算機に**影響を与えない**（別系統）。

**Non-Goals:**
- イベント周回ドロップ率の自前計測。Atlas データ＋手入力で代替。
- レイド・ポイント走り・ミッション最適化（Phase 2+）。
- 素材計算機（`/material`）との自動合算（Phase 2）。
- バトルシミュ・3T編成。

## Decisions

### 1. ボックス層は貪欲法/単純計算、周回層は既存LPソルバに委譲
**Rationale**: ボックス層は決定論（N 箱 × 固定値）なので最適化不要。交換所の「通貨を素材へ最適配分」が必要な場合も、候補が小規模なため `lib/material-selection-advisor.ts` と同じ貪欲法で厳密解が出る（実績ある軽量パターン）。周回層だけが真の最適化問題で、ここは `solve()` をそのまま使う。

**solve() への接続方法**: イベント通貨・イベント素材を「アイテム」、各周回ノードを「クエスト」とみなした `Drops` 互換オブジェクトを `lib/event-plan.ts` で合成し、需要（必要通貨数・必要素材数）を制約として `solve()` に渡す。ソルバは item_id 空間に依存しないため、イベント通貨に専用の一時 item_id を割り当てれば既存ロジックがそのまま動く。

**代替案**: ソルバにイベント概念（箱・在庫上限）を直接組み込む案は、`/farming` の安定した恒常パスにイベント固有の制約を持ち込みリグレッションリスクが高い。アダプタ層で `Drops` に正規化して**ソルバを汚さない**方を選ぶ。

### 2. イベントデータは別 KV キー・別取り込み経路（恒常データと分離）
**Rationale**: 恒常データ（`all_drops_json`、`get-drops.ts`）は短縮ID安定化など繊細な不変条件を抱える（`docs` / id_registry 参照）。イベントは寿命が短く更新頻度・スキーマが異なるため、`event_data_json` として独立させ、`getWars` の `eventId == 0` フィルタは現状維持する。これにより既存パスへのリグレッションをゼロにする。

**取り込み対象（Phase 0 で実フィールド確定）**: Atlas `nice_event/{eventId}` から抽出。
- `lotteries[].cost.item`(通貨アイテム) ＋ `lotteries[].cost.amount`(1抽選あたり通貨数)。1箱の通貨コスト = Σ`boxes[].maxNum`(同 `boxIndex` 内) × `cost.amount`。
- `lotteries[].boxes[]`: `boxIndex`(箱ラウンド)・`gifts[].objectId`/`gifts[].num`・`maxNum`(箱内個数)・`isRare`。1箱の中身 = `boxIndex` でグルーピングし `num`×`maxNum` を集計。
- `shop[]`: `purchaseType=='item'` を素材交換として採用。報酬 = `targetIds`×`setNum`、対価 = `cost.item`+`cost.amount`、在庫 = `limitNum`。
- 周回ドロップ: 周回ノード（type='event'/afterClear='repeatLast'）の `/nice/JP/quest/{id}/{phase}` `drops[]`（`objectId`/`num`/`runs`/`dropCount`）から 1周期待値を算出。

開催中＋直近終了のイベントのみを対象に GH Actions 定期実行（日次想定。箱内容は会期中不変だが、周回ドロップのサンプルは増えるため日次更新で精度が上がる）。

### 3. イベント周回ドロップは「Atlas データ優先・手入力フォールバック」
**Rationale**: ロト箱内容・交換所・箱コストは Atlas に会期開始時点から存在し信頼できる。一方、**1周あたりのイベント通貨/素材ドロップ数**はゲームデータの固定報酬＋RNG ドロップの合算で、新規イベントでは計測データが揃わない。MVP では (a) Atlas のクエストドロップデータがあれば既定値として採用し、(b) 無い/不正確な場合に備え「周回ノードと1周あたり通貨/素材」をユーザーが上書き入力できる欄を必ず用意する。コミュニティは会期序盤に最適ノードと通貨/AP をすぐ確立するため、手入力でも実用に足る。

**代替案**: ドロップ計測を自前で行う案はスコープ過大（別プロジェクト級）。Atlas＋手入力で MVP を成立させ、計測連携は将来検討。

### 4. UI 配線は `/events` 新設（既存導線を変えない）
**Rationale**: 恒常の `/material` → `/farming` 導線は完成度が高く、ここにイベントを混ぜると複雑化する。イベントは寿命と思考プロセスが別物（「箱を割る」発想）なので独立ルート `/events`（一覧）→ `/events/[id]`（計画）とする。ダッシュボード（既に `DashboardEvent` を表示）から該当イベントの計画ページへ導線を張る。

### 5. ヒーロー体験は「既存の永続ロスターの不足素材 → 箱数」。育成ゴールは新規入力せず既存資産を読む
**前提（実装済み資産）**: 育成ゴールと所持の永続ロスターは**既に存在する**。
- `hooks/create-chaldea-state.ts` の `ChaldeaState`（`{ [servantId]: { disabled, targets: { ascension/skill/appendSkill: { disabled, ranges:[{start,end}] } } } }`）＝サーヴァントごとの育成目標。
- `useChaldeaState` が localStorage `'material'` に保存し、`use-cloud-sync.ts` でデバイス間同期（`sync` spec）。所持数も同経路で永続化。
- `lib/get-materials.ts`（目標→必要素材）／`lib/diff-materials.ts`（必要−所持＝不足）で不足素材が既に算出可能。

**Rationale**: したがって「何箱割れば育成がどれだけ進むか」のための育成ターゲットを**新規入力させる必要はない**。イベントプランナーは既存の `ChaldeaState` ＋所持数から不足素材を読み、「このイベントの箱/交換所が不足をどれだけ埋めるか・必要箱数・残りはフリクエ何周か」を提示する。これは旧 US-8（/material 連携）に相当するが、**重い新規実装ではなく既存資産への配線**であり、ユーザー指摘の通り「方向性の整理」で MVP に入れられる。

**実装方針**: イベントプランナーから `useChaldeaState`／`get-materials`／`diff-materials` を読み取り専用で利用 → 不足素材を需要に変換 → ボックス確定報酬・交換所での充当を計算（育成インパクト）→ 残需要を `solve()`（Decision 1）で周回計画化。**書き込みは行わない**（ロスター/所持の更新は既存 `/material` の責務のまま。リグレッション面を read-only に限定して抑える）。

**代替案（却下）**: イベントページ内に自己完結の育成ターゲット入力を別途作る案は、既に完成している永続ロスターの二重持ちになり UX もデータも分断するため却下。

**留意**: ログイン無し／ロスター未設定のユーザー向けに、ロスターに依存しない素朴な「欲しい素材を直接指定」フロー（US-1 の簡易版・US-2/US-3）もフォールバックとして残す。

### 6. イベント種別はボックス（ロト）に集中。レイドは同ソルバで次段
**Rationale**: ユーザーの詳細ユースケースがボックス特化であり、ボックスは「箱→通貨→周回」の構造が最も明快。レイドは「討伐数→通貨/素材→周回」で**最後の周回層は同じ `solve()` に乗る**が、討伐数・ドロップの取り込みとUIが別物なので Phase 2（US-10）に切る。MVP のデータ型・アダプタ（`lib/event-plan.ts`）はレイドを後付けできるよう、「需要（通貨/素材）→ ノード周回」の汎用形を保つ。

## Risks / Trade-offs

### [Risk 1] イベント周回ドロップ率データの欠如・不正確（Phase 0 で大幅に低減）
- Phase 0 検証結果: イベント周回ノード（type='event' / afterClear='repeatLast'）の `/nice/JP/quest/{id}/{phase}` `drops[]` に `objectId`/`num`/`runs`/`dropCount` が揃い、1周期待値 = `dropCount/runs × num` で通貨・素材とも算出可能（実例: 通貨≈54/周, AP40）。よって**会期がある程度進んだイベントでは Atlas データで足りる**。
- 残存影響: 新規イベントの会期序盤はサンプルが少なく欠落/不正確。
- Mitigation: Decision 3 の手入力フォールバック（US-5）を会期序盤向けに**維持**。算出結果には「データソース（Atlas/手入力）」と概算である旨を明示。Atlas データが後追い更新されたら自動反映。

### [Risk 2] Atlas `nice_event` のスキーマ変動・地域差（JP/NA で内容差）
- 影響: パース失敗・誤データ。
- Mitigation: 取り込みスクリプトに Zod 等のバリデーションを入れ、未知形状は当該イベントをスキップ（既存 `lib/master-data/validation.ts` の DRY_RUN 検証思想を踏襲）。リージョンは既存 `constants/atlasacademy.ts` の `region` を踏襲（JP）。

### [Risk 3] 確定報酬と RNG ドロップの二重計上
- 影響: ボックス確定報酬と周回ドロップを足すと過大評価になりうる。
- Mitigation: 「ボックス確定分」と「周回ドロップ分」を出力で明確に分離表示。需要充足判定は両者を合算するが内訳を必ず提示。

### [Risk 4] 会期終了後のデータ寿命
- 影響: 終了イベントの KV が肥大／古い情報の表示。
- Mitigation: 取り込み対象を「開催中＋終了後 N 日」に限定し、それ以降は KV から落とす。UI でも会期外は明示。

## Open Questions（決定済み）
- **レアボックス（`isRare`）**: 決定 → レアボックスの中身が**素材なら充当計算に含める**。**礼装/サーヴァント/コマンドコードは「N箱で確定入手」として別バッジ表示**し、素材の充当計算とは分離する。
- **交換所の在庫上限（`limitNum`）**: 決定 → 充当は**`limitNum` まで**で打ち切り、超過分は警告表示して残需要として周回（ソルバ）に回す。
- **入力UIの主従**: 決定 → **主＝既存ロスターの不足素材（US-1）**、**副＝目標箱数 / 欲しいアイテム数の直接指定（US-2、ロスター非依存フォールバック）**。
