## Context

現在の `HistoryGraph` は `recharts` の `type="category"` を使用しており、データポイントの間隔が時間の経過と比例していません。また、表示期間の絞り込みや将来の予測機能が欠けています。

## Goals / Non-Goals

**Goals:**
- X軸を時間軸（タイムライン）として正しく表現する。
- ユーザーが表示期間（1ヶ月〜1年）を切り替えられるようにする。
- 現在の進捗ペースに基づいた目標達成予想日（AP/周回数が0になる日）を視覚化する。
- ダッシュボードと履歴ページで重複しているグラフ・ロジックを共通化する。

**Non-Goals:**
- より複雑な統計モデル（季節性など）の導入。
- サーバーサイドでのフィルタリング・集計（今回はフロントエンドでの処理に留める）。

## Decisions

### 1. Recharts XAxis の設定
- **決定**: `XAxis` の `type` を `number` に設定し、`dataKey` に UNIX タイムスタンプを使用する。
- **理由**: 時間の経過を正確な距離で表現するため。`domain={['auto', 'auto']}` と `tickFormatter` を組み合わせることで、動的な期間表示に対応できる。

### 2. 達成予測ロジック
- **決定**: 最小二乗法を用いた単回帰分析（Simple Linear Regression）を採用する。
- **理由**: 実装が軽量であり、現在の減少トレンドを直線的に伸ばすという目的には十分であるため。
- **詳細**: 選択された期間内の全データポイントを使用し、傾き $m$ と切片 $b$ を算出。$Y=0$ となる $X = -b/m$ を達成予想日とする。

### 3. X軸の表示範囲（ウィンドウ）の動的調整
- **決定**: 未来方向に表示期間の 20%〜30% 程度の「バッファ領域」を常に確保する。
- **理由**: 予測線のトレンドを視覚化しつつ、遠すぎる予測によって過去データが圧縮されるのを防ぐため。
- **スケーリング**:
  - 達成予想日が `現在日 + 表示期間` 内にある場合：予想日をグラフに収めるように X軸を拡張する。
  - 達成予想日がそれより遠い場合：X軸は `現在日 + (表示期間 * 0.3)` 程度でクリップし、予測線は端でフェードアウトまたはラベル表示を行う。

### 4. コンポーネントの共通化
- **決定**: `components/farming/FarmingHistoryChart.tsx` を新規作成し、ダッシュボードと履歴ページの両方から利用する。
- **理由**: ロジックの重複を防ぎ、将来的なメンテナンス性を向上させるため。

### 5. 表示期間のフィルタリング
- **決定**: フロントエンドの `useMemo` でデータをフィルタリングする。
- **理由**: 既存の API (`/api/farming/history`) が最新 50 件を返しているため、現状のデータ量であればクライアント側で十分高速に処理可能。

## Risks / Trade-offs

- **[Risk] データ不足による予測の不安定さ** → **[Mitigation]** データポイントが 2 点未満の場合や、傾きが正（APが増えている）場合は予測線を表示せず、「データ不足」などのメッセージを表示する。
- **[Risk] 50件制限による長期グラフの精度** → **[Mitigation]** 1年などの長期表示においてデータが50件では足りない可能性がある。将来的に API の `limit` を拡張可能にする。
