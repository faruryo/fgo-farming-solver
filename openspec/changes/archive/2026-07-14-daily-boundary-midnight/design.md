## Context

`lib/todo/period.ts` はデイリーの日界を `DAILY_RESET_OFFSET_MS = 4h`（リセット 4:00 JST・期限 3:59 JST）でモデル化しているが、実ゲームのデイリーミッションは 0:00 JST 開始・締切。daily-notification-evening-threshold で通知開始を 22:59 JST にしたことで、0時以降の配信が実締切後になる問題が顕在化した。

## Goals / Non-Goals

**Goals:**
- デイリーの期間境界を実ゲームどおり 0:00 JST にする（期限 23:59 JST）。
- 通知開始を 21:59 JST（期限2時間前）にし、就寝前かつ実締切前に届くようにする。

**Non-Goals:**
- ウィークリー境界（月曜 0:00 JST）・イベント・カスタムの変更。
- cron 頻度の変更。
- 過去の day key / notification_log の移行（不要: ID 体系は不変、旧レコードは無害）。

## Decisions

### 1. `DAILY_RESET_OFFSET_MS` を 4h → 0 に変更
period.ts の境界計算は offset を引くだけの構造なので、定数変更のみで day key・deadline・次リセット時刻がすべて 0:00 基準になる。ヘッダコメントと境界テストを更新する。

### 2. `THRESHOLD_MS.daily` を 5h → 2h に変更
期限 23:59 に対し通知開始 21:59 JST。窓幅2時間は実測のバッチ間隔（通常1〜1.5h、まれに2.7〜3.5h）に対し取りこぼし小リスクがあるが、「23時頃に届いてほしい」というユーザー希望との баланスで**ユーザーが選択**した値。取りこぼしが実際に問題になったら閾値を広げる。

## Risks / Trade-offs

- **[Risk]** 窓2時間 < 観測最大間隔3.5h のため、長い実行間隔が窓を完全に跨ぐとその日のデイリー通知が飛ばない。
- **[Mitigation]** ユーザー了承済みのトレードオフ。発生頻度は低く（2h超の間隔は稀）、実害があれば閾値を3hに広げる後続変更で対応。
- **[Risk]** デプロイが 0:00〜3:59 JST を跨ぐ場合、直後のバッチで day key が+1日ずれ、新しい日のデイリー通知が（前日分送信済みでも）新 key として送られ得る。
- **[Mitigation]** 新 key は実ゲーム上も有効な「今日の」デイリーであり誤報にはならない。1回限りの過渡事象として許容。
