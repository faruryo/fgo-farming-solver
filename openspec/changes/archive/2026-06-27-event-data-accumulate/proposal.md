## Why

`event-planner` の終了済みイベントシミュレート（復刻予習）機能（`ap-budget-quartz-cost` で導入）は、計画ページが KV (`event_data_json`) のイベントデータを読むことで成立する。しかし現状の取り込みジョブ（`scripts/refresh-event-data.ts`）は `ENDED_GRACE_DAYS = 30` で「開催中＋終了後30日以内」のロト型イベントだけを**毎回ゼロから作り直して**保存する。結果:

- 終了後30日を超えた過去ロトは KV から消える → 終了済みシミュレートを実データで試せない。
- ロト未開催かつ直近終了も無い期間（年の大半）は KV が空 → `/events` に何も出ず、機能を検証できない（現にいま空）。

過去のロト型イベントを恒久的に蓄積保持すれば、いつでも過去イベントで終了済みシミュレート（復刻予習・振り返り）ができ、機能の実データ検証も可能になる。

## What Changes

- **取り込みを「全置換」から「蓄積マージ」に変更**
  - 既存 KV `event_data_json` を `event.id` キーの map に読み込み、今回フェッチした分で upsert（同一 id は新データ優先＝Atlas 修正反映）、取り込み窓に入らない過去 id は**削らず温存**する。
  - 出力は `startedAt`（または `endedAt`）でソートして保存。
  - ロト型イベントは年1〜3件と少ないため、蓄積による blob 肥大は KV 25MB 上限に対し十分小さく許容範囲。
- **過去バックフィルの手段を追加**
  - 前向きマージだけでは「既に30日超で消えた／一度も取り込まれていない過去ロト」は戻らない。`basic_event.json`（過去全件を含む）から対象範囲を遡って過去ロトの `nice_event` を取得し一括投入する初期シード手段を用意する（例: 取り込み対象の開始時刻下限を渡す手動実行）。
- **既存の安全弁は維持**
  - `basic_event.json` が空（上流障害）なら書き込み拒否。
  - 該当ロト0件時は既存 KV を温存して正常終了（no-op）。マージ化後は「新規追加0件でも既存を壊さない」を満たす。

### Modified Capabilities
- `event-planner`: 「イベントデータの取り込み」要件を、開催中＋直近に限らず**過去のロト型イベントを蓄積保持（マージ）**する挙動へ改定。過去バックフィル・既存温存・上流障害保護のシナリオを定義する。

## Impact

- **実装**: `scripts/refresh-event-data.ts` の対象フィルタと書き込みをマージ化。バックフィル用の対象範囲指定（環境変数 or CLI 引数）を追加。
- **ワークフロー**: `.github/workflows/refresh-event-data.yml` に手動バックフィル導線（`workflow_dispatch` 入力でシード範囲を指定）を追加。
- **データ/KV**: `event_data_json` が累積的に増加（年1〜3件）。スキーマ（`EventData = { events, updatedAt }`）は不変。読み取り側（`lib/get-events.ts`・`/events`・計画ページ）は無改修で動作。
- **非破壊**: 恒常ドロップ `all_drops_json` には影響しない。

## Non-goals

- 過去イベントのドロップ率精度改善はしない（取り込めるものをそのまま蓄積）。
- `/events` の終了済みグループが履歴蓄積で長くなる件の UI 改善（年次グルーピング・折りたたみ等）は本 change のスコープ外（将来別 change）。
- 恒常ドロップデータ・`getWars` の `eventId == 0` フィルタは変更しない。
