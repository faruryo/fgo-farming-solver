## ADDED Requirements

### Requirement: アイテム入手による残りの減少（目標固定再ソルブ）
システムは、比較スナップショット以降にユーザーがアイテムを入手したことで「残りに必要なAP・周回数」がどれだけ減ったかを、目標を現在に固定した再ソルブで算出し表示しなければならない (SHALL)。目標を両辺とも現在値に固定するため、目標を増やしても本指標は増加してはならない (SHALL NOT)。

#### Scenario: アイテム入手による減少量の算出
- **WHEN** 現在の目標（`material/result`）と、現在の所持（`posession`）および比較スナップショットの過去所持が取得できるとき
- **THEN** `need = max(0, 現在目標 − 所持)` を所持別に作り、現在所持版と過去所持版をそれぞれソルバーで解く。
- **THEN** 減少AP `= solve(現在目標 − 過去所持).total_ap − solve(現在目標 − 現在所持).total_ap`、減少周回 `= 同 total_lap`、減少費用 `= round(減少AP / 144 / 168 * 10000)` を表示する。

#### Scenario: 目標増加が減少量に影響しないこと
- **WHEN** 比較スナップショット以降に育成目標が増えた（`material/result` の総数が増加した）とき
- **THEN** 目標は両辺とも現在値で固定されているため、目標増加そのものは減少AP・減少周回を増減させない。

#### Scenario: 過去所持が欠損している場合のフォールバック
- **WHEN** 比較スナップショットに `posession` が記録されていないとき
- **THEN** 減少AP・減少周回・減少費用は算出せず非表示とし、育成総量や新規サーヴァント等の他指標のみ表示する。

### Requirement: 最古スナップショットを基準とした単一比較
システムは、期間タブを設けず、存在するスナップショットのうち最も古いもの（古い順に 1ヶ月前 → 1週間前 → 前回の優先で、実比較できる最古の期間）を1つだけ基準として選び、それと比較した進捗（育成総量、アイテム入手による残りAP/周回数/費用の減少、サーヴァント成長）を表示しなければならない (SHALL)。比較基準の時点は見出しの上に明示し、「前回」は時点が曖昧なため経過日数（昨日 / N日前）で表示しなければならない (SHALL)。

#### Scenario: 最古の存在スナップショットを採用
- **WHEN** 1ヶ月前・1週間前・前回いずれにも実スナップショットが存在するとき
- **THEN** 最も古い「1ヶ月前」のスナップショットを基準として比較し、「1ヶ月前と比べて」と数値の上に表示する。

#### Scenario: 古い期間が存在しない場合のフォールバック
- **WHEN** 1ヶ月前のスナップショットが存在せず、1週間前（または前回）のみ存在するとき
- **THEN** 存在する中で最も古い期間（1週間前→前回の順）を基準として比較する。

#### Scenario: 前回基準の時点表示
- **WHEN** 基準が「前回」スナップショット（最近の1件）であるとき
- **THEN** 経過時間から「昨日」「N日前」のように相対日数で時点を表示する。

### Requirement: 結果ページの計算履歴グラフ表示
システムは、ソルバー結果ページ（`/farming/results/[id]`）において、進捗報告パネルの代わりに計算履歴の消費AP推移グラフを表示しなければならない (SHALL)。

#### Scenario: 履歴グラフの表示
- **WHEN** ログイン済みで計算履歴が2件以上あるとき
- **THEN** 結果ページに計算履歴の消費AP推移グラフ（`/api/farming/history` 由来）を表示する。
- **THEN** モード別スタッツ（周回数・消費AP・費用）はそのまま表示する。

#### Scenario: 履歴不足・未ログイン時
- **WHEN** 未ログイン、または計算履歴が2件未満のとき
- **THEN** 履歴グラフは表示しない（自動的に非表示）。

### Requirement: 新規入手サーヴァントの検出
システムは、比較スナップショット間で新規に「所持」化されたサーヴァントを検出し、新規入手を歓迎するメッセージの選択に用いなければならない (SHALL)。検出は、比較対象スナップショットに chaldea state（material）が記録されている場合にのみ行い、記録が無い場合は新規 0 件として扱わなければならない (SHALL)。

#### Scenario: ガチャ後の進捗評価
- **WHEN** 比較スナップショット間で chaldea state の `disabled` が `true` から `false` に変化したサーヴァントが存在するとき
- **THEN** 新規入手サーヴァントとして検出し、歓迎メッセージの選択に用いる。
- **THEN** 目標を現在に固定して算出するため、新規サーヴァントの必要素材は現在目標に含まれ、アイテム入手による減少指標を歪めない。

#### Scenario: 比較対象に chaldea state が無い場合
- **WHEN** 比較対象スナップショットに chaldea state（material）が存在しない（`null`）とき
- **THEN** 新規サーヴァントを 0 件として扱う。
- **THEN** 全所持サーヴァントを新規と誤判定しない。

## MODIFIED Requirements

### Requirement: 進捗報告画面の表示
システムは、ユーザーの育成進捗を報告するマシュ進捗パネルを、ダッシュボードの「あなたの育成進捗」セクション（達成率円グラフの上）に表示しなければならない (SHALL)。比較対象が無い場合も破綻せず案内を表示しなければならない (SHALL)。

#### Scenario: ダッシュボードでの進捗表示
- **WHEN** ダッシュボードを開き、現在の育成状態と進捗データが取得できたとき
- **THEN** 「あなたの育成進捗」セクション上部にマシュ進捗パネルが表示され、下部に既存の達成率円グラフが表示される。

#### Scenario: 比較対象なし時の表示
- **WHEN** 過去スナップショットが存在しないか、指定期間のスナップショットが取得できないとき
- **THEN** 「登録お疲れ様」モードのメッセージとともにパネルが表示される。

### Requirement: total_ap 進捗の tier 判定
システムは、経過時間に対する自然回復AP（5分=1AP）を基準とし、「アイテム入手による残りAPの減少量（`reducedAp`）」を 4 段階に分類しなければならない (SHALL)。

#### Scenario: 大進捗の判定
- **WHEN** `reducedAp >= naturalRecoveryAp * 1.5` のとき
- **THEN** tier は `large` と判定される。

#### Scenario: 中進捗の判定
- **WHEN** `naturalRecoveryAp <= reducedAp < naturalRecoveryAp * 1.5` のとき
- **THEN** tier は `medium` と判定される。

#### Scenario: 小進捗の判定
- **WHEN** `0 < reducedAp < naturalRecoveryAp` のとき
- **THEN** tier は `small` と判定される。

#### Scenario: 進捗ゼロの判定
- **WHEN** `reducedAp <= 0` かつ育成総量が 0 のとき
- **THEN** tier は `none` と判定される。

### Requirement: サーヴァント成長度の差分表示
システムは、chaldea state の育成目標レンジ（再臨/スキル/アペンド）が前回スナップショットから縮んだ（=実際に育成した）場合、その成長量の合計（育成総量）と主なサーヴァントを進捗として表示しなければならない (SHALL)。

#### Scenario: 育成総量の表示
- **WHEN** 比較スナップショット間でサーヴァントの育成目標レンジ合計が縮んだとき
- **THEN** 縮んだ量の合計を「育成総量」として表示し、寄与の大きいサーヴァントを「○○ を ○ レベル育成しました」のように一覧表示する。

## REMOVED Requirements

### Requirement: 期間別進捗比較
**Reason**: 前回/1週間前/1ヶ月前の3期間タブは情報過多だったため廃止。存在する最古のスナップショット1件を基準とする「最古スナップショットを基準とした単一比較」に置き換えた。
**Migration**: 期間タブによる切り替えは廃止。比較基準は `selectBaseline`（古い順: 1ヶ月前→1週間前→前回）が1件だけ自動選択する。

### Requirement: 新規入手サーヴァントによる必要AP増加分のオフセット
**Reason**: 進捗指標が「目標を現在で固定した再ソルブ」に変わり、新規サーヴァントの必要素材は現在目標に含まれるため、rarity 別推定APによるオフセット計算自体が不要になった。
**Migration**: rarity-AP テーブルによるオフセット加算は廃止。新規サーヴァントは「新規入手サーヴァントの検出」要件として、検出のみ行い歓迎メッセージに用いる。

### Requirement: 目標増加のポジティブ表示
**Reason**: アイテム入手による減少指標が目標を現在で固定して算出するようになり、目標増加は減少指標に影響しなくなったため、目標増加分を独立して表示・補正する必要がなくなった。先日の `targetApIncrease` 単位取り違えバグの上位対処。
**Migration**: 目標増加に関する独立表示・補正ロジックは廃止。進捗は「育成総量」と「アイテム入手による残りの減少（目標固定再ソルブ）」で表現する。
