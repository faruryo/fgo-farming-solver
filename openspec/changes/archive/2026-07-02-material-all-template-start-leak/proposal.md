## Why

育成素材計算機 (`/material`) で、多くのサーヴァントのアペンドスキル（および霊基再臨・スキル）の現在値 (`start`) が、意図せず古いデフォルト値（例: アペンドスキル全枠 `1`）のまま表示され続けるバグが確認された。本番相当のブラウザ `localStorage['material']` を確認したところ、460 体中 457 体でアペンドスキル 5 枠すべてが `start: 1`（正しいデフォルトは `start: 0`）になっていた。

根本原因は 2 つの要素の組み合わせである。

1. `chaldeaState.all`（共通目標テンプレート）の `targets.*.start` は、`all` エントリが最初に生成された時点の値のまま恒久的に凍結されている（`start` を編集する UI が存在しないため）。
2. `hooks/use-local-storage.ts` の永続化 effect は、マウント時に `mergeChaldeaState` でマージした**フル state（全サーヴァント分の明示エントリ）を、ユーザー操作なしでも `localStorage` に自動的に書き戻す**。このため、一度でも `/material` を開いたユーザーの `localStorage` には、実際には誰も編集していないサーヴァントも含め全員分の個別エントリが既に存在し、その `start` は書き戻された時点の（凍結された）`all.start` 由来の値で固定されている。

この 2 つ目の性質により、「`mergeChaldeaState` が `all` から `start` を継承しないようにする」だけでは、**既に個別エントリとして書き戻し済みの既存データは直らない**（`mergeChaldeaState` の `...state` スプレッドが `all` 由来の値より優先されるため）。したがって本変更は、単に継承経路を断つだけでなく、**未所持（`disabled: true`）サーヴァントの `start` を毎回正しいデフォルト値へ矯正するロジック**を追加する。未所持サーヴァントは、そもそも `start` を編集する UI 手段が存在しない（所持済みにするまで現在レベルのスライダー/チップは操作できない）ため、この矯正は既存の正当なユーザー入力を破壊しない。

この `all → 個別サーヴァント` への `start` 漏洩は、`createServantState()` のデフォルト値が過去 3 回変更された際（コミット `ad73129`, `7db1a48`, `26bd9cf`）にも毎回同じ症状で再発しており、対症療法的な修正が繰り返されてきた。今回はこの伝播経路と、既に書き戻し済みの汚染データの両方に対処する。

## What Changes

- `mergeChaldeaState`（`hooks/use-chaldea-state-merger.ts`）が `state.all` から個別サーヴァントへ `targets` を複製する際、`end`（目標値）のみを `all` から継承し、`start`（現在値）は常に `createServantState()` の正しいデフォルト値を用いるように変更する（新規サーヴァント追加時の再発防止）。
- 加えて `mergeChaldeaState` に、**`disabled: true`（未所持）のサーヴァントについては、マージ後の `targets.*.ranges[].start` を毎回正しいデフォルト値へ強制的に上書きする**ロジックを追加する。これにより、既に `localStorage` に書き戻し済みの汚染データ（452/460 体が該当）も次回ロード時に自動的に是正される。
- `disabled: false`（所持済み）のサーヴァントで、かつ全枠が凍結値と一致している少数のケース（観測データでは 5/460 体）は、意図的な自動修正の対象外とする。ユーザーは各アペンド/スキルチップの長押しまたは右クリック（-1 操作、下限クランプで `1→0`）により、既存 UI から 1 アクションで手動修正できる。
- 既存ユーザーの `localStorage['material']` に対する専用マイグレーションスクリプトの追加は行わない。上記の「未所持サーヴァントは毎回矯正」ロジック自体が、次回ページロード時に自動適用されるため個別マイグレーションは不要。

## Capabilities

### New Capabilities
(none)

### Modified Capabilities
- `material`: 「共通目標設定」要件のうち、`chaldeaState.all` から個別サーヴァントへの `targets` 複製仕様（`end` のみ継承）と、「所持状態のトグル」要件に、未所持サーヴァントの `start` を常に正しいデフォルトへ矯正する挙動を追加する。

## Impact

- `hooks/use-chaldea-state-merger.ts`: `mergeChaldeaState` の `state.all` 分岐ロジックの変更、および未所持サーヴァントの `start` 矯正ロジックの追加。
- `hooks/use-chaldea-state-merger.test.ts`: 既存の回帰テストの一部（`all` テンプレートの `start` がそのまま複製されることを前提にしたテスト）を新しい期待値に更新し、未所持サーヴァントの矯正・所持済みサーヴァントの非破壊を検証するテストを追加。
- `openspec/specs/material/spec.md`: 「共通目標設定」および「所持状態のトグル」要件の該当シナリオを更新。
- **既知のトレードオフ**: 過去に所持していた（`start` を編集した）サーヴァントを未所持に戻すと、`start` は編集済みの値ではなく正しいデフォルトへリセットされる（未所持サーヴァントの現在値は元々「育成完了判定・所持数カウントの対象外」であり、この挙動は既存仕様と整合する）。この経路は個別トグルだけでなく `ms-servants-io.tsx` の一括インポート欄を空にする操作（全サーヴァント一括で未所持化）も含む。design.md の Risks で詳述する。
- 影響範囲は `/material` ページの `chaldeaState` 初期化ロジックのみ。`/material/result` や素材計算 (`sumMaterials`) 自体には手を入れない。
