## Why

期限間近プッシュ通知の判定窓が「期限−閾値 から1時間」のワンショットであるため、GitHub Actions の cron 実行間隔に生じる穴（実測で1.5〜3.5時間）に窓が入ると通知が永久に送られない。実害として `notification_log` の全履歴はデイリー2件のみで、ウィークリー通知は機能導入以来一度も配信されていない（2026-07-12 の weekly-2026W28 は未完了だったが窓 [11:59, 12:59 JST) に実行が無く欠落）。

## What Changes

- **判定窓の拡張**: `isDueForNotification` の送信条件を `dueAt <= now < dueAt + 1h` から `dueAt <= now < deadline` に変更する。重複送信は `notification_log` への `INSERT ... ON CONFLICT DO NOTHING` が既に防止しているため、窓を広げても同一タスクに二重送信は発生しない。
- **通知文言の調整**: 実行遅延時に「残り3時間です」等の固定文言が実残時間とずれるため、「期限が近づいています」系の非定量的な文言に変更する。
- **cron 頻度は変更しない**（GitHub Actions 予算制約）。

## Capabilities

### New Capabilities

*(なし)*

### Modified Capabilities

- `todo-notifications`: 「期限間近のプッシュ通知 (Deadline Near Notifications)」要件の配信条件を、閾値時刻ちょうどのワンショット窓から「閾値時刻以降〜期限まで」に変更し、通知文言を非定量的な表現に修正する。

## Impact

- **配信バッチ**: `scripts/send-todo-notifications.ts` の `isDueForNotification` と通知タイトル文字列。
- **既存ユーザーへの挙動変化**: これまで欠落していたウィークリー通知・遅延時のデイリー/イベント/カスタム通知が届くようになる。期限超過後は送らないため、通知が突然増えるのは「本来送られるべきだったもの」に限られる。
- **デプロイ直後の1回性の影響**: デプロイ後最初のバッチ実行で、現在窓内（dueAt を過ぎ期限前）の未送信タスクにまとめて通知が飛ぶ可能性があるが、`notification_log` により各タスク1回のみ。
- **フォローアップ候補（別 change）**: デイリー通知の窓 [00:59, 03:59 JST) が全域深夜帯であり、遅延時は最悪3時台の配信になる。閾値を早めて夕方〜夜帯に移す、またはデイリー通知を廃止する検討を残す。
