## Context

期限間近プッシュ通知は GitHub Actions の毎時 cron で `scripts/send-todo-notifications.ts` を実行して配信する。現行の `isDueForNotification` は `dueAt = deadline - threshold` に対し `dueAt <= now < dueAt + 1時間` のワンショット窓で判定しており、「毎時必ず1回実行される」前提に立っている。しかし GitHub Actions のスケジュール実行は遅延・欠落が常態で（実測で実行間隔1.5〜3.5時間）、窓に実行が1回も入らないと通知は永久に送られない。実績として weekly 通知は一度も配信されたことがない。

## Goals / Non-Goals

**Goals:**
- バッチ実行がいつ来ても、期限前である限り未送信の期限間近通知が必ず送られるようにする。
- 同一タスクへの二重送信は引き続き発生させない。

**Non-Goals:**
- cron 頻度・ワークフロー構成の変更（GitHub Actions 予算制約のため据え置き）。
- クライアント側（トグル・購読管理）の変更。
- 閾値（デイリー3h / ウィークリー12h / イベント・カスタム24h）自体の見直し。

## Decisions

### 1. 判定窓を「閾値時刻から期限まで」に拡張
`isDueForNotification` を `now >= dueAt && now < deadlineMs` に変更する。

- ワンショット窓は「毎時実行される」前提でのみ成立する設計であり、実行基盤がその前提を満たさない以上、窓側で吸収する。
- 重複防止は送信直前の `notification_log` への `INSERT ... ON CONFLICT DO NOTHING ... RETURNING`（アトミックなクレーム）が既に担っており、窓の広さは送信回数に影響しない。ワンショット窓は冗長な二重ガードだった。
- 上限を期限に取ることで、期限超過後の無意味な通知（もう間に合わない）は送らない。

### 2. 通知文言を非定量的な表現に変更
以下に変更する。なお送信時に `deadline - now` から動的に「残り約N時間」を計算すれば正確な残時間表示は可能だが（ずれるのは現行の事前固定文言のみ）、通知の価値は「期限前に気づかせること」にあり、簡潔さを優先して静的文言を採用する。

- デイリー: 「デイリーミッションの期限が近づいています！」
- ウィークリー: 「ウィークリーミッションの期限が近づいています！」
- イベント: 「[イベント名] の交換期限が近づいています！」
- カスタム: 「[タスク名] の期限が近づいています！」

## Risks / Trade-offs

- **[Risk]** デプロイ後最初の実行で、現在窓内（dueAt 超過・期限前）の未送信タスクにまとめて通知が飛ぶ。
- **[Mitigation]** `notification_log` により各タスク1回のみ。対象は「本来送られるべきだった通知」に限られ、実害より回復に近い。
- **[Risk]** 期限直前（例: 残り10分）に初回実行が来た場合でも通知が飛び、ユーザーが間に合わない可能性がある。
- **[Mitigation]** 「期限が近づいています」という文言は残時間を約束しないため誤報にはならない。期限超過後は送らないので無意味な通知は発生しない。
- **[Risk]** 重複排除は `notification_log` の PK `(subscription_id, notification_key)` による購読単位のため、endpoint がローテートする再購読（ブラウザ再 subscribe で一般的）では新しい `subscription_id` になり、窓内の未期限タスクが再送される。窓拡張によりこの再送可能期間が 1時間 → 閾値幅（最大24時間）に拡大する（同一 endpoint の再購読は `ON CONFLICT(endpoint) DO UPDATE` が id を据え置くため再送なし）。
- **[Mitigation]** 新 endpoint ごと・タスクごとに1回で自己限定的な既存の設計的負債であり許容する。恒久策（dedup キーを `(user_id, notification_key)` にする案）はマルチデバイス配信の意味論が変わるためスコープ外。
- **[Risk]** デイリーの実効窓は [00:59, 03:59 JST) と全域が深夜帯にあり、cron 遅延時の配信は旧設計の最悪 1時台から**最悪 3時台まで**遅れ得る（UX 後退）。クワイエットアワーで抑制すると窓全体が死ぬため回避不能。
- **[Mitigation]** 根本原因はデイリーの閾値・窓タイミング自体であり本 change の Non-Goal。フォローアップ候補（デイリー通知の閾値を早めて夕方〜夜帯へ移す、またはデイリー通知の廃止）を proposal に記録して別 change とする。
