# Design: progress-5tier-lap-value

## Context

本番実データ(user 1183…8814、baseline 2026-06-06 → 2026-07-06)での実測が設計の根拠:

| 指標 | 値 | 現行tier |
|---|---|---|
| LP限界換算(現行・クランプ済) | 7,192 AP (自然回復比 0.84x) | small |
| 素材単価の独立AP換算 | 19,222 AP (2.2x) | — |
| 周回換算(育成目標のみ) | 888周 (29.6周/日) | — |
| **周回換算(バッファ込み)** | **1,261周 (42.0周/日)** | — |
| 労力=全獲得の周回換算 | 2,755周 (91.8周/日) | — |

発見された問題(いずれも実測で確認):

1. **LPの限界価格割引**: ソルバーは全不足をまとめて最適化するため、各素材の増分は「他素材のついで」の限界価格で評価される。独立単価合計19,222APに対し限界7,192APと2.7倍の乖離。目標が大きいアカウントほど割引が強まり、育てるほど褒められにくい逆インセンティブ。
2. **バッファ非参照**: `effectiveRequired`(育成必要数+ストックバッファ)は quest-efficiency の D3 で「全farming画面が共有」と定めた定義だが、進捗tierだけ育成目標のみを参照。バッファ埋め周回(+373周相当)が「余剰=0点」扱い。
3. **単位の問題**: 労いの文脈での労働量は AP ではなく周回数が実態(1周の手間はクエストAPに依らずほぼ一定)。低AP帯しか解放していない新米は同じ自然APでより多く周回するため、周回換算は新米を自然に持ち上げる。

ペルソナ検証(受け入れ基準): P1やる気ある新米=large到達可能 / P3ユーザー通常イベント月=large / P3ボックス・レイド月=legendary / P4常時りんご勢=large〜legendary / P5備蓄王=tierは補完でも労力表示で認知される。ログイン勢はアプリ対象外とし保護しない。

## Goals / Non-Goals

**Goals:**
- tierをアカウント状態(在庫量・目標規模・クエスト解放度)に依らず「働いた月は働いたと出る」指標にする
- ボックス/レイド全力月だけが到達する最上位段階を設ける(5段階)
- バッファ込み実効不足で quest-efficiency と目標定義を統一(D3整合)
- 労力(全獲得)と方向性(不足直結)を別軸で認知する

**Non-Goals:**
- QP・種火・絆など posession に映らない周回の評価(既知の穴として残す)
- サーバAPI・スナップショット形式の変更
- rarity-APテーブル関連要件の整理(tier経路では未使用になるが本changeでは触らない)
- gross(同一素材の獲得と消費の両建て)の復元 — net観測の制約は継続

## Decisions

### D1: tier主指標 = バッファ込み実効不足の解消ぶんの周回換算(方向性)

```
単価_i   = 1 / (選択クエスト内の最高drop_rate_i)   ※無ければ全クエスト中の最高率
実効目標_i = material/result_i + (stockEnabled ? buffer(item_i) : 0)
adjustedNow_i = max(現在所持_i, 過去所持_i)          ※消費クランプ(継承)
前進周回 = Σ_i (max(0,実効目標_i−過去所持_i) − max(0,実効目標_i−adjustedNow_i)) × 単価_i
perDay  = 前進周回 / (elapsedMinutes / 1440)
```

- 独立換算にすることでLPの限界価格割引を廃し、不足素材を掘る限り傾き≈1を構造的に保証。
- 単価をユーザーの選択クエスト内で解くことで、進捗度(解放状況)の差を吸収(新米は率が低い=1個が重い=高く労われる)。選択外にしかドロップが無い素材は全クエスト率へフォールバック(新米のquests未整備対策)。
- 入手経路は問わない(ストーリー報酬・イベント交換も所持増として満額計上)。モチベーション装置として意図した挙動。
- 代替案「LP限界換算+閾値を0.5xへ調整」: 傾きがアカウント状態の関数のままで、閾値がいたちごっこになるため却下。
- 代替案「自然回復AP比の正規化を維持(AP独立換算)」: 周回換算のほうが労働の実態に近く、新米ブーストが単価に内蔵されるため周回を採用。

### D2: 5段階しきい値(1日あたり推定周回数)

| tier | 条件 | 想定 |
|---|---|---|
| `legendary`(新設) | ≥ 60周/日 | ボックス/レイド全力月 |
| `large` | ≥ 15周/日 | 自然APフル活用〜りんご投入(新米の低AP帯なら自然APのみで≈19周/日) |
| `medium` | ≥ 5周/日 | コツコツ継続 |
| `small` | > 0 | 触った月 |
| `none` | = 0 | 補完判定へ |

- アンカー: 15周/日 ≈ 自然回復288AP/日を使い切る周回量。実測でユーザー通常イベント月42周/日=large、ボックス月は銅銀素材の大量不足解消で60超=legendary。
- 名称 `legendary` は仮。実装時に i18n キーとセットで確定(ユーザー承認)。

### D3: 労力軸 = 全獲得素材の周回換算(育成投入は含まない)

- `労力周回 = Σ_i max(0, 現在所持_i − 過去所持_i) × 単価_i`(余剰込み・QP除外)。
- 育成投入は労力ではない(育成の認知は既存の育成総量・サーヴァント成長表示が担う)。消費はどの軸にも乗せず中立。
- 表示: 「この期間の活動」に「推定活動量 約N周相当」を追加。
- マシュのメッセージは2軸で修飾: 方向性tier高×労力高=効率も量も称賛 / tier低×労力高=備蓄・活動を労う / 両方低=従来どおり。

### D4: tier=0(前進なし)時の補完

- 労力周回に同じラダーを適用して tier を補完する。ただし `legendary` は方向性限定とし、補完では `large` を上限とする(備蓄王が毎イベント最上位になるインフレ防止)。
- 全指標ゼロは従来どおり `zero_progress`。育成総量・新規入手による労いメッセージ優先も従来どおり。

### D5: 表示「目標への前進」は周回主役・AP併記

- 主数値「+N周ぶん前進」、内訳に AP相当(独立AP換算 = Σ解消×最安AP単価)と費用(`yenFromAp` 継承)。
- LP再ソルブ(`compute-reduction.ts` のソルバー経路)は表示からも外れるため削除候補。`reducedLap`(周回最小プラン)も役目を終える。
- tierと表示数値が同一指標になり、「largeなのに前進が小さい」不整合が構造的に消える。

### D6: バッファ解決は quest-efficiency の既存実装を共有

- `resolveStockBuffer(stored, legacySurplus)` + `effectiveRequired(item, required, buffer, enabled)` を `lib/quest-efficiency.ts` から import(重複実装しない)。
- localStorage: `efficiency/stockBuffer` / `efficiency/stockEnabled` / legacy `efficiency/surplusThreshold` を読む。stockEnabled=OFF のユーザーはバッファ0(=育成目標のみ)で従来相当。

## Risks / Trade-offs

- [独立換算は「ついで取り」を満額計上するため、LP換算より常に甘い] → 労い装置としては意図した方向。厳密な残りプラン短縮は結果ページ/履歴グラフが担う。
- [`ProgressTier` 拡張が既存の switch/map を静的に壊す] → TypeScript の網羅性チェックで検出可能。mashu-messages・演出・i18n を同changeで揃える。
- [単価の全クエストフォールバックは未解放クエストの率で安く値付けする] → ストーリー報酬ブーストと相殺方向。許容。
- [同一素材のfarm→consumeのgrossは相殺されたまま] → net観測の既知の制約(継承)。
- [しきい値5/15/60は実測1アカウント+ペルソナ想定による初期値] → 定数を一箇所に集約し調整可能にする。

## Migration Plan

クライアント算出のみでデータ移行なし。`ProgressTier` に値追加のためサーバの暫定 `'none'` は互換。ロールバックはコード revert のみ。
