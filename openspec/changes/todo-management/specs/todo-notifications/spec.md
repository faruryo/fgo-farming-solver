## ADDED Requirements

### Requirement: 自動 TODO 追加 (Auto TODO Generation)
システムは、ユーザー向けに以下の標準TODOタスクを自動的に生成しなければならない (SHALL)。
1. デイリーミッション: 毎日午前3:59 JSTに期限切れ/リセット。
2. ウィークリーミッション: 毎週日曜日午後11:59 JSTに期限切れ/リセット。
3. イベント交換ショップ: 開催中のイベントの交換期限終了時刻に期限切れ。

自動生成されるタスクは、該当する期間の開始時に初期化され、未完了（未チェック）の状態で作成される。

#### Scenario: デイリーミッションの自動生成
- **WHEN** ユーザーがダッシュボードまたはTODO管理画面を表示し、前回のログイン日と異なる新しいJST日付（午前4:00基準）が開始しているとき
- **THEN** 「デイリーミッションをクリアする」というタスクが自動生成される。

#### Scenario: ウィークリーミッションの自動生成
- **WHEN** ユーザーがダッシュボードまたはTODO管理画面を表示し、新しい週（月曜日午前0:00 JST基準）が開始しているとき
- **THEN** 「ウィークリーミッションをクリアする」というタスクが自動生成される。

#### Scenario: イベント交換ショップタスクの自動生成
- **WHEN** 開催中のイベントが存在し、そのイベントのアイテム交換ショップの終了期限（マスターデータ `NiceEvent.shopEnd`）が設定されているとき
- **THEN** 「[イベント名] アイテム交換を完了する」というタスクが自動生成され、期限にイベント交換終了日時が設定される。

### Requirement: 自動追加の ON/OFF 設定 (Auto-Add Settings)
システムは、ユーザー設定画面において、自動 TODO 生成およびプッシュ通知の有効/無効をカテゴリ（デイリー、ウィークリー、イベント交換）ごとに個別にトグルで切り替える設定項目を提供しなければならない (SHALL)。

#### Scenario: 自動生成の無効化
- **WHEN** ユーザーが設定画面で「デイリーミッション自動追加」を無効（OFF）に設定したとき
- **THEN** 以降のデイリーミッション TODO は自動生成されない。また、現在未完了として存在する自動生成済みのデイリーミッション TODO は非表示または削除される。

#### Scenario: 設定の同期
- **WHEN** ユーザーが設定を変更したとき
- **THEN** 設定値は localStorage の `todoSettings` キーに保存され、既存のオートシンク機構を通じて Cloudflare KV に同期される。

---

### Requirement: プッシュ通知の許諾と登録 (Push Notification Permission and Subscription)
システムは、ブラウザの Web Push API を使用して、ユーザーのブラウザから通知許可を取得し、プッシュ通知の購読情報をサーバーに登録しなければならない (SHALL)。

#### Scenario: プッシュ通知の有効化
- **WHEN** ユーザーが設定画面で「プッシュ通知を有効にする」ボタンをクリックしたとき
- **THEN** ブラウザが通知許可を求め、許可された場合は Service Worker 経由で PushSubscription を作成する。
- **THEN** 作成された購読情報（Endpoint, Keys）をサーバーの API `/api/notifications/subscribe` へ送信し、D1 データベースに保存する。

#### Scenario: 購読情報の失効処理
- **WHEN** サーバーがプッシュ通知を送信した際に、ブラウザ側の購読が失効している（410 Gone などのエラーが返る）とき
- **THEN** サーバーは D1 データベースから該当する PushSubscription レコードを削除する。

---

### Requirement: 期限間近のプッシュ通知 (Deadline Near Notifications)
システムは、完了していない自動生成 TODO の期限が近づいた場合、Web Push 通知をユーザーに配信しなければならない (SHALL)。
通知を配信する閾値は以下とする。
- デイリーミッション: 期限切れの3時間前（毎日午前1:00 JST）
- ウィークリーミッション: 期限切れの12時間前（毎週日曜日午前12:00 JST）
- イベント交換ショップ: 期限切れの24時間前

#### Scenario: デイリーミッション期限間近の通知配信
- **WHEN** 配信バッチ（GitHub Actions ワークフロー等の定期実行）が午前1:00 JSTに実行され、デイリーミッション TODO が未完了かつプッシュ通知が有効なユーザーの購読情報が存在するとき
- **THEN** 「デイリーミッションの期限が残り3時間です！」という通知が配信される。

#### Scenario: ウィークリーミッション期限間近の通知配信
- **WHEN** 配信バッチが日曜日午前12:00 JSTに実行され、ウィークリーミッション TODO が未完了かつプッシュ通知が有効なユーザーの購読情報が存在するとき
- **THEN** 「ウィークリーミッションの期限が残り12時間です！」という通知が配信される。

#### Scenario: イベント交換期限間近の通知配信
- **WHEN** 配信バッチが実行され、イベント交換 TODO の期限が残り24時間未満であり、かつ未完了のとき
- **THEN** 「[イベント名] の交換期限が残り24時間です！」という通知が配信される。

---

### Requirement: 手動チェックと永続化 (Manual Task Check-off and Sync)
ユーザーは、自動生成された TODO および手動で追加した TODO の完了状態（チェックボックス）を切り替えることができ、その状態はクラウド間で同期されなければならない (SHALL)。

#### Scenario: タスクの完了チェック
- **WHEN** ユーザーが TODO アイテムのチェックボックスをオン（完了）にしたとき
- **THEN** 該当タスクのステータスが「完了」に変更され、localStorage が更新され、オートシンク経由で `/api/cloud` に送信される。
- **THEN** 期限間近になってもこのタスクに関するプッシュ通知は送信されない。
