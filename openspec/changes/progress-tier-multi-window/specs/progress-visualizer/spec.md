## MODIFIED Requirements

### Requirement: 進捗 tier の判定
システムは、進捗の達成度を `legendary` / `large` / `medium` / `small` / `none` の 5 段階（tier）で判定しなければならない (SHALL)。主指標は「バッファ込み実効不足の解消ぶんの推定周回数（前進周回）」を経過日数でならした値（周/日）とする。

前進周回は次で算出しなければならない (SHALL): 素材ごとに 単価（1個あたり推定周回数）= 1 ÷ ユーザー選択クエスト内の最高ドロップ率（選択クエスト内にドロップが無い素材は全クエスト中の最高率にフォールバック）、実効目標 = 育成目標の必要数 +（ストックバッファが有効なら バッファ量、無効なら 0）（quest-efficiency 機能の実効必要数と同一の定義）、現在所持は過去所持で下限クランプ（消費中立）した上で、実効不足の過去値と現在値の差分に単価を掛けて合算する。入手経路（周回・イベント交換・ストーリー報酬）は問わない。

しきい値（周/日）: `≥100 → legendary`、`≥30 → large`、`≥15 → medium`、`>0 → small`、`0 → none`（補完判定へ）。前進周回が 0 の期間は、労力周回（獲得素材の周回換算）に同じしきい値を適用して tier を補完しなければならない (SHALL)。ただし補完による tier は `large` を上限とし、`legendary` は前進周回によってのみ到達できる (SHALL)。

#### Scenario: 周回換算による判定
- **WHEN** 前進周回の周/日が算出できたとき
- **THEN** `≥100 → legendary`、`≥30 → large`、`≥15 → medium`、`>0 → small` と判定する。

#### Scenario: バッファ込み実効不足への前進
- **WHEN** 育成目標は充足済みだが、所持がストックバッファ込みの実効目標に満たない素材を獲得したとき
- **THEN** その獲得分も前進周回に計上する（ストックバッファ機能が無効なユーザーはバッファ0として従来の育成目標のみで判定する）。

#### Scenario: 育成消費が tier を下げないこと
- **WHEN** 同一期間内に素材の獲得と育成による消費の両方が発生したとき
- **THEN** 消費は前進周回を減少させず、獲得のみが tier 判定に寄与する。

#### Scenario: 選択クエストにドロップが無い素材の単価
- **WHEN** 不足解消した素材のドロップ元がユーザーの選択クエストに含まれないとき
- **THEN** 全クエスト中の最高ドロップ率で単価を算出し、0 扱いにしない。

#### Scenario: 前進ゼロ時のスループット補完
- **WHEN** 前進周回が 0 で、獲得素材（余剰込み）の労力周回が正のとき
- **THEN** 労力周回に同じしきい値を適用して tier を補完する。ただし `legendary` には到達させない（上限 `large`）。

#### Scenario: 進捗ゼロの判定
- **WHEN** 前進周回・労力周回・育成総量・新規入手のいずれも無いとき
- **THEN** tier は `none`、フォールバックは進捗ゼロと判定する。

### Requirement: 30/60/90日候補のうちperDay最大の窓を基準とした比較
システムは、期間タブを設けず、比較を「**今（ライブ状態）**」と「**過去スナップショットの複数候補**」の2点で行わなければならない (SHALL)。候補は「30日前」「60日前」「90日前」それぞれに時刻差が最小のスナップショットとし（同一スナップショットに複数候補が解決した場合は1つに畳む）、サーヴァント育成データも所持データも持たない degenerate スナップショットは候補から除外しなければならない (SHALL NOT baseline として採用)。

採用する baseline は、各候補について前進周回を周/日に正規化した値（forwardPerDay）のうち最大のものとする (SHALL)。forwardPerDay が正となる候補が1つも無い場合は、各候補の労力周回を周/日に正規化した値（effortPerDay）のうち最大のものを採用する (SHALL)。前進周回と労力周回の指標は跨いで比較してはならない (SHALL NOT)（forwardPerDay が正の候補が1つでもあれば、effortPerDay がより大きい他候補があっても採用しない）。複数候補の値が同値の場合は、より短い窓（30日 > 60日 > 90日の順）を優先する (SHALL)。

比較基準の時点は見出しの上に経過日数（`昨日` / `N日前`）で明示しなければならない (SHALL)。

#### Scenario: 直近30日にバースト獲得があれば30日窓が採用される
- **WHEN** 30日窓のforwardPerDayが60日窓・90日窓のforwardPerDayより大きいとき
- **THEN** 30日窓の baseline・tier・表示値を採用する。

#### Scenario: 直近30日に入手経路のバーストが乗らなかった場合は長い窓が採用される
- **WHEN** デイリーミッションチケットの一括交換等で獲得がまとまっており、直近30日の窓では未消化のためforwardPerDayが低いが、60日窓または90日窓で見るとforwardPerDayがより高いとき
- **THEN** forwardPerDayが最大となる窓（60日窓または90日窓）の baseline・tier・表示値を採用する。

#### Scenario: 全窓で前進ゼロなら労力周回で比較する
- **WHEN** 30/60/90日いずれの候補もforwardLapsが0以下のとき
- **THEN** 各候補のeffortPerDayのうち最大の候補を採用し、tierを補完する（legendaryには到達させない）。

#### Scenario: 直近のみ存在する場合は最古を候補とする
- **WHEN** スナップショットが全て30日前より新しいとき（例: 今日が 06-07 で 06-02 / 06-03 / 06-06）
- **THEN** 30/60/90日いずれの候補も最も古い 06-02 に解決し、1候補として扱う。

#### Scenario: 中身の無いスナップショットを基準にしない
- **WHEN** 候補がサーヴァント育成データも所持データも持たない degenerate レコードのとき
- **THEN** それを候補から除外し、比較に使える中身を持つ他の候補を基準とする。比較できるものが無ければ初回登録等のフォールバックを表示する。

#### Scenario: 変化ゼロでも実比較は成立する
- **WHEN** 採用された baseline と今で数値変化が無いとき
- **THEN** 「スナップショット無し」ではなく「変化ゼロ（登録お疲れさま）」として扱う。
