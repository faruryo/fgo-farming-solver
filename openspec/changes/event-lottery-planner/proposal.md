## Why

本プロダクトの固有の強みは **最小AP / 最小周回の周回ソルバ**だが、現状その対象は恒常フリークエストのドロップ（`MASTER_DATA`）に限定されている。一方、FGO プレイヤーの周回需要のピークは**イベント、とりわけ「ロト（ボックスガチャ）」**にある。「あと何箱割れるか」「目標の箱数に必要なイベント通貨と周回数」「イベント交換所で何を取るか」は、イベントのたびに全プレイヤーが手計算している領域だ。

最大の競合 Chaldea はこのイベント/ロト計画を厚く持っており、ここが乗り換え障壁になっている。しかし重要なのは、**この問題は本プロダクトの既存LPソルバがほぼそのまま使える**点である。ロト周回は「目標箱数 → 必要イベント通貨 → その通貨を最小AP/最小周回で集める」という構造で、最後の層は `lib/solver.ts` の `solve()`（需要アイテム＝イベント通貨）に帰着する。「ソルバの質」で殴り合える数少ない土俵であり、開発者自身がイベントごとに使う＝ドッグフーディングが効く機能でもある。

関連する見送り判断（データ自動取り込み）は `docs/decisions.md` を参照。本提案はそれとは独立で、ゲーム連携不要・既存の Atlas Academy データ取得基盤の延長で成立する。

## User Stories

ペルソナは FGO のボックス（ロト）イベントを周回するマスター。中心的なジョブは、コスト計算そのものよりも一段上の「**自分の育成ゴールから逆算して『何箱割るか』を決める**」という目標設定にある。やみくもに箱を割るのではなく、「この箱を割れば狙っている育成がどこまで進むか」を見て、割るべき箱数を意思決定したい。

> 既存資産の前提: 「誰を・どこまで育てるか」の育成ゴールと所持数は、**既に永続ロスターとして実装済み**（`hooks/create-chaldea-state.ts` の `ChaldeaState`、localStorage `'material'` ＋ `use-cloud-sync` でデバイス間同期、`get-materials`/`diff-materials` で不足素材算出）。本機能はこれを**読み取って再利用**するのが核で、育成ターゲットを新規入力させない。

> イベント種別について: 詳細を詰めるのはまず**ボックス（ロト）イベント**。レイドイベントは構造が異なる（ボス討伐数 → 通貨/素材）が同じ周回ソルバに乗るため、**次のイベント種別**として Phase 2 で扱う（US-10）。

### MVP（Phase 1 で満たす）

- **US-1 既存ロスターの不足素材 → 箱数目標設定（ヒーロー）**
  マスターとして、**既に登録済みの育成ゴール（永続ロスター）の不足素材**に対して、「このボックスイベントを何箱割れば不足がどれだけ埋まり、残りはフリクエ何周か」を見ながら目標箱数を決めたい。育成ゴールから逆算して「割るべき箱数」を意思決定したいから。
  受け入れ: 既存 `ChaldeaState` ＋所持数から不足素材を読み（書き込みはしない・read-only）、ボックス確定報酬・交換所が不足をどれだけ充当するか（育成インパクト）を可視化し、必要箱数・必要通貨・必要周回数・消費AP（果実/林檎換算）を提示する。**新規の育成ターゲット入力は作らず既存資産を再利用**する（旧 US-8 を「重い新規実装」ではなく既存資産への配線として MVP に取り込む）。（spec: 育成ゴールからの箱数目標設定 / ロト計画の算出）

- **US-2 箱数からコスト試算 / 欲しい素材の直接指定（ロスター非依存フォールバック）**
  マスターとして、「△箱割る」と先に箱数を決めているとき、または未ログイン・ロスター未設定のときは、**箱数 or 欲しい素材を直接指定**してコストを試算したい。
  受け入れ: 入力＝目標箱数 または 欲しい素材と個数。出力＝必要通貨（所持分を差し引き）・周回数・消費AP・確定獲得素材。永続ロスターに依存しない。

- **US-3 確定報酬の内訳プレビュー**
  マスターとして、「△箱割ると確定でどの素材が何個もらえるか」を一覧で見たい。**ロトだけで足りる素材**と**別途フリクエが要る素材**を切り分けたいから。
  受け入れ: ボックス確定報酬の累計を表示。確定分と周回ドロップ分を内訳分離。（spec: 確定報酬と周回ドロップの内訳分離）

- **US-4 所持通貨を反映した残り周回**
  周回途中のマスターとして、**今持っているイベント通貨**を入力して、残りの必要周回数だけを知りたい。会期中の進捗確認に使うから。
  受け入れ: 所持通貨を需要から差し引いて算出。

- **US-5 ドロップ手入力で会期序盤から使える**
  新規イベント序盤のマスターとして、ドロップデータが未整備でも、**自分が周回するノードの1周あたり通貨/素材を手入力**して計算したい。会期開始直後こそ計画したいから。
  受け入れ: Atlas データが無い場合は手入力欄を提供。使用したドロップ値のソース（Atlas/手入力）を明示。（spec: ドロップデータのフォールバックと明示）

- **US-6 交換所の在庫上限の警告**
  マスターとして、交換所で欲しい数が**在庫上限を超える**場合に警告してほしい。取りきれない素材を別手段（フリクエ等）で確保する判断をしたいから。
  受け入れ: `limitNum` 超過分を警告表示し、上限で打ち切る。（spec: 欲しいアイテム数からの必要箱数逆算）

- **US-7 イベント一覧と導線**
  マスターとして、**開催中のボックスイベントを一覧**で見て計画ページに入りたい。ダッシュボードの該当イベントからも飛びたい。
  受け入れ: `/events` 一覧 → `/events/[id]`。ダッシュボードからの導線。（spec: ロト計画 UI）

### 将来（Phase 2 以降、本 MVP のスコープ外）

- **US-9 AP予算からの上限**: ライト勢として、手持ちの林檎/果実（AP予算）から、現実的に何箱割れるかを逆算したい。
- **US-10 レイドイベント対応**: レイドイベントのマスターとして、討伐数・1討伐あたり通貨/素材から最適な周回計画を立てたい（同じソルバに乗せる、次のイベント種別）。
- **US-11 ポイント走り/ミッション**: イベントポイント報酬・ミッション達成の最適化。

> 注: 旧 US-8（素材計算機＝永続ロスターとの統合）は、その基盤（`ChaldeaState`＋同期＋`get-materials`/`diff-materials`）が既に実装済みのため、新規 Phase 2 機能ではなく **US-1 として MVP に取り込んだ**。

## What Changes

**MVP（Phase 1）はロト型イベントに絞る。** 恒常素材計算（`/material`）・既存ソルバ（`/farming`）には一切変更を加えない。

- **イベントデータの取り込み（新規）**
  - Atlas Academy の `nice_event/{eventId}`（ロトボックス内容・箱コスト・交換所・ポイント報酬）を取得し、コンパクト化して KV に保存する。取得はワーカーではなく GitHub Actions 定期ワークフローで行う（既存 `refresh-nice-war.yml` と同じ「重い取得は GH Actions、ワーカーは KV を読むだけ」方針。無料プランの CPU kill 回避）。
  - 現状 `lib/get-wars.ts` は `eventId == 0` でイベントを除外している。本変更ではイベントを**別系統**として取り込み、既存の恒常データ（`all_drops_json`）は汚さない。
- **ロト計画 UI（新規 `/events` 配下）**
  - 開催中（および直近終了）のロト型イベント一覧。
  - 1イベントを選び、**入力**: 目標箱数 _または_ 欲しいロト/交換アイテムの個数、所持イベント通貨、周回ノード（と1周あたり通貨/素材）。
  - **出力**: 必要イベント通貨総量、ボックス確定報酬の累計（素材獲得見込み）、目標達成に必要な周回数・消費AP（既存ソルバを再利用）、果実/林檎換算。
- **周回層は既存ソルバを再利用**
  - イベント通貨を「需要アイテム」として `solve()` に渡す。イベント周回ノードのドロップ（通貨＋素材）を `Drops` 互換の一時データとして合成して投入する。ソルバ本体は無改修。

## Non-goals

- **レイド／高難易度／ポイント走り／ミッション最適化**: Phase 2 以降。MVP はロト＋交換所のみ。
- **イベント素材を恒常素材計算（`/material`）に自動合算**: 統合は Phase 2。MVP は独立ページ。
- **イベント周回ドロップ率の自前収集**: 新規のドロップ計測基盤は作らない。Atlas のドロップデータ（あれば）＋ユーザー手入力フォールバックで成立させる（後述リスク参照）。
- **バトル/3T編成シミュレーション**: スコープ外（Chaldea Laplace 相当は作らない）。
- **ゲーム本体との自動連携・所持データ自動取り込み**: `docs/decisions.md` の通り見送り済み。

## Capabilities

### New Capabilities
- `event-planner`: イベント（ロト型）データの取り込みと、目標箱数/交換数から必要通貨・周回数・消費AP・確定報酬を算出するロト計画機能。

### Modified Capabilities
- なし（MVP は既存 capability を改変しない。`solver` は外部から再利用するのみ）。

## Impact

- **データ取り込み**: 新規 `scripts/refresh-event-data.ts`（Atlas `nice_event` → コンパクト KV）＋新規 GH Actions ワークフロー（開催中イベントを対象に日次程度）。新 KV キー（例 `event_data_json`）。
- **データ層**: `lib/master-data/types.ts` にイベント型（`EventLottery` / `EventLotteryBox` / `EventShopItem` / `EventCurrency`）追加。新規 `lib/get-events.ts`。
- **ソルバ連携**: 新規 `lib/event-plan.ts`（箱→通貨→`Drops` 合成→`solve()` 呼び出しのアダプタ。ソルバ本体は無改修）。
- **UI**: 新規 `app/events/page.tsx`（一覧）・`app/events/[id]/page.tsx`（計画）＋ `components/events/`。
- **i18n**: `locales/{ja,en}.json` にイベント計画関連キー追加。
- **モック**: `mocks/` にイベントデータのサンプル（dev 検証用）。
- **既存への影響**: なし（`all_drops_json` / `solve()` / `/material` / `/farming` は不変）。
