## ADDED Requirements

### Requirement: スクリーンショット複数枚アップロード導線
システムは、既存の所持数編集 UI（`PossessionModal` 等）から、マイルーム所持アイテム一覧のスクリーンショットを複数枚まとめて選択・投稿できる入口を提供しなければならない (SHALL)。画像はブラウザ内で処理され、外部サーバーへ送信してはならない (SHALL NOT)。

#### Scenario: 複数枚の同時選択
- **WHEN** ユーザーが所持数編集 UI から「スクリーンショットから取り込む」を選択したとき
- **THEN** ファイル選択ダイアログ（複数選択可）またはドラッグ&ドロップで、複数枚の画像を一度に指定できる。

#### Scenario: 複数回に分けての追加投稿
- **GIVEN** ユーザーがすでに1枚以上の画像を投稿済みで、解析結果を確認する前の状態である
- **WHEN** ユーザーが追加でスクリーンショットを投稿したとき
- **THEN** 既存の投稿分を破棄せず、追加分を含めて解析対象に加える。

#### Scenario: 対応外ファイルの拒否
- **WHEN** 画像形式でないファイル（例: `.txt`, `.pdf`）が選択されたとき
- **THEN** そのファイルは解析対象から除外され、対応外である旨がユーザーに通知される。
- **THEN** 他の有効な画像ファイルの処理は継続される。

### Requirement: アイテム識別（品目名テキスト認識を主軸、アイコン照合は補助）と数量抽出
システムは、投稿された各スクリーンショットから、品目名テキストの文字認識と既知アイテム名辞書へのファジーマッチを主たる手段としてアイテム候補（Atlas ID）を求めなければならない (SHALL)。アイコン画像とのパーセプチュアルハッシュ照合は、品目名認識の結果を検算する補助的な信号として用いてよい (MAY) が、識別の必須条件としてはならない (SHALL NOT)。識別されたアイテムに隣接する数値領域を光学文字認識して所持数候補を抽出しなければならない (SHALL)。認識処理はクライアント側で完結しなければならない (SHALL)。

#### Scenario: 品目名認識のみで高信頼に識別できる
- **WHEN** 品目名テキストのファジーマッチが既知アイテム名に高い類似度で一致したとき
- **THEN** アイコン照合の結果に関わらず、そのアイテムは高信頼の候補として扱われる。

#### Scenario: アイコン照合が品目名認識の結果を裏付ける
- **WHEN** 品目名テキストのファジーマッチとアイコン領域のパーセプチュアルハッシュ照合が同一の Atlas ID を指したとき
- **THEN** そのアイテムはより高信頼の候補として扱われる。

#### Scenario: アイコン照合と品目名認識が食い違う
- **GIVEN** 品目名ファジーマッチとアイコン照合がそれぞれ異なる Atlas ID を指した
- **WHEN** 候補を確定しようとしたとき
- **THEN** どちらの Atlas ID も自動採用せず、その領域は「要確認」としてレビューUI上に両方の候補とともに表示される。

#### Scenario: 品目名認識が低信頼、またはできなかった場合
- **GIVEN** 品目名のファジーマッチ類似度が閾値付近、または文字がつぶれて既知アイテム名に一致しなかった
- **WHEN** アイコン照合のみが既知アイテムに一致したとき
- **THEN** アイコン照合による Atlas ID を「要確認」の候補として扱い、レビューUI上で他の候補と区別して表示される。

#### Scenario: 数量の抽出
- **GIVEN** アイテムが（高信頼または要確認として）識別された
- **WHEN** そのアイテムに隣接する数値テキスト領域を光学文字認識したとき
- **THEN** 認識された数値が、そのアイテムの所持数候補として記録される。

#### Scenario: 識別はできたが数量が読み取れない
- **WHEN** アイテムは識別できたが、隣接する数値領域から数値を抽出できなかったとき
- **THEN** そのアイテムは所持数候補なしとしてレビューUIに表示され、ユーザーが手動で数値を入力できる。

#### Scenario: 桁区切り文字の誤認識を正規化する
- **GIVEN** 光学文字認識が桁区切りのカンマをピリオドと誤読した（例: 「3,052」を「3.052」と認識）
- **WHEN** 所持数候補を数値化するとき
- **THEN** カンマ・ピリオドのいずれも桁区切りとして正規化してから数値化し、「3.052」を「3」と誤って解釈してはならない (SHALL NOT)。

#### Scenario: どちらの方式でも識別できない
- **WHEN** 画像中の領域が、アイコン照合・品目名ファジーマッチのいずれによっても既知アイテムに一致しなかったとき
- **THEN** その領域は候補として抽出されず、所持数の更新対象にもならない。

### Requirement: 解析結果のレビューと確定
システムは、抽出した所持数候補を一覧表示し、ユーザーが個別に確認・修正・除外したうえで確定操作を行うまで `posession` を更新してはならない (SHALL NOT)。

#### Scenario: 候補一覧の表示
- **WHEN** 投稿されたスクリーンショットの解析が完了したとき
- **THEN** 認識されたアイテムごとに、アイコン・アイテム名・現在の `posession` 値・抽出された所持数候補が一覧表示される。

#### Scenario: 差分の表示
- **WHEN** 候補一覧が表示されたとき
- **THEN** 各候補は、アイテム名と、現在の `posession` 値から候補値への差分（変更前→変更後）が分かる形で表示される。
- **THEN** 現在値からの変動幅が大きい候補、または信頼度が低い／矛盾ありの候補（閾値は実装フェーズで定める）は、他の候補より目立つ形で区別され、一覧の並び順でも優先的に表示される。

#### Scenario: 認識元クロップ画像のオンデマンド確認
- **GIVEN** 候補一覧に、変動幅が大きい・信頼度が低い・矛盾ありのいずれかで目立つ表示になっている候補がある
- **WHEN** ユーザーがその候補を展開操作したとき
- **THEN** その候補の抽出元となった元画像のクロップ（カード領域）が表示され、光学文字認識の結果と元画像を見比べて確認できる。
- **THEN** 通常（目立つ表示になっていない）候補では、クロップ画像は既定で表示されない。

#### Scenario: 候補値の個別修正
- **WHEN** ユーザーが一覧上のいずれかの候補値を編集したとき
- **THEN** 確定時にはユーザーが編集した値が採用される。

#### Scenario: 候補の個別除外
- **WHEN** ユーザーが一覧上のいずれかの候補を除外操作したとき
- **THEN** そのアイテムは確定操作の対象から外れ、`posession` の値は変更されない。

#### Scenario: 確定操作
- **WHEN** ユーザーがレビューUIで「反映する」を選択したとき
- **THEN** 除外されていない各候補の値で `localStorage['posession']`（Atlas ID キー）が上書きされる。
- **THEN** レビューUIは閉じ、通常の所持数編集 UI の表示にも即時反映される。

#### Scenario: キャンセル
- **WHEN** ユーザーがレビューUIで「反映する」を選択せずに閉じたとき
- **THEN** `posession` はいかなる変更も受けない。

#### Scenario: 認識結果ゼロ件
- **WHEN** 投稿されたすべてのスクリーンショットからアイテム候補が1件も抽出されなかったとき
- **THEN** レビューUIは「認識できたアイテムがありません」と表示し、確定操作は無効化される。

### Requirement: 複数画像間での重複・矛盾の統合
複数のスクリーンショットに同一アイテムが写っている場合、システムは重複する候補を1件に統合し、値が食い違う場合はユーザーに選択させなければならない (SHALL)。

#### Scenario: 同一値での重複
- **GIVEN** 2枚以上のスクリーンショットから同一アイテムの候補が抽出され、いずれも同じ所持数を示している
- **WHEN** 候補一覧を生成したとき
- **THEN** そのアイテムは1件の候補として表示される。

#### Scenario: 異なる値での重複（矛盾）
- **GIVEN** 2枚以上のスクリーンショットから同一アイテムの候補が抽出され、所持数が食い違っている
- **WHEN** 候補一覧を生成したとき
- **THEN** そのアイテムは「矛盾あり」として表示され、いずれの値を採用するかユーザーが選択（または手動入力）するまで、候補値は自動的に決定されない。

### Requirement: 画像端で見切れたカードの扱い
所持アイテム一覧はスクロール可能なリストであり、1枚のスクリーンショットの上端・下端でカードが部分的にしか写らないことがある。システムは、画像の上端または下端に接しており全体が写っていないと判定されるカード領域を、通常の（全体が写っている）カードと区別して扱わなければならない (SHALL)。

#### Scenario: 見切れたカードは低信頼として扱われる
- **GIVEN** カード領域が画像の上端または下端に接しており、カードの一部（アイコン・品目名・所持数のいずれか）が画像内に収まっていない
- **WHEN** そのカード領域を解析したとき
- **THEN** 抽出された候補（識別できた場合・所持数が部分的に読めた場合を含む）は「要確認」として扱われ、自動的には確定候補にならない。

#### Scenario: 見切れたカードが他の画像の全体表示と重複する
- **GIVEN** ある画像で見切れて「要確認」となったカードと同一のアイテムが、別の画像では全体表示されている
- **WHEN** 候補を統合するとき
- **THEN** 全体が写っている方の読み取り結果が優先され、見切れた方の読み取り結果によって上書きされない。

### Requirement: 既存の所持数更新経路との共存
スクリーンショット取り込みは、既存の手入力・JSON貼り付け（`MsItemsIo` 等）による所持数更新と共存し、いずれの経路を使っても `posession` のスキーマ・保存先（`localStorage['posession']`、Atlas ID キー）は変わらない (SHALL)。

#### Scenario: 取り込み後の他機能への反映
- **WHEN** スクリーンショット取り込みで `posession` が更新されたとき
- **THEN** 育成記録モードの消費/返還計算、`/material/result` の所持数表示、クラウド同期は、手入力時と同様に即座に整合する。

#### Scenario: 取り込み対象外アイテムの値は維持される
- **WHEN** スクリーンショット取り込みで一部アイテムのみ候補が確定されたとき
- **THEN** 候補に含まれなかった他のアイテムの `posession` 値は変更されない。
