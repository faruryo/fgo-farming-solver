## Context

`docs/decisions.md`（2026-06-16）では、所持アイテムの自動取り込み（スクショOCR / HTTPSスニッフィング）を「保守コストが高い」「ページング地獄」を理由に見送っている。今回のユーザーストーリーは複数枚投稿を前提としており、ページング地獄には対応済みだが、保守コスト・精度の懸念は依然として残るため、実装方式の選定でこれを最小化する必要がある。

本プロダクトは Cloudflare Workers 上でホストされ、サーバー側APIコストに慎重（[[project_updater_workers]] 参照: 定期ジョブは GH Actions、cron worker は全廃）。ユーザーからは「サーバー費用ゼロのクライアント側完結」方式が明示的に選ばれている。よって、マルチモーダルLLM画像直読みは今回のスコープ外とし、ブラウザ内で完結する文字認識（OCR）+ アイコン照合を採用する。

**実測スパイクの結果（設計に反映済み）**: 実装前に実際のマイルーム所持アイテム一覧スクリーンショットで簡易検証を行った。
- `tesseract.js`（jpn.traineddata）は、ゲーム内フォント（縁取り・エンボス調）に対して二値化前処理を挟んでも数字はほぼ、アイテム名はほぼ全滅という結果だった。
- macOS純正 Vision framework（参照値・ブラウザには使えない）は前処理なしのカード全体クロップで名前・所持数ともほぼ完璧に読めた。これにより「画像自体がOCR困難」ではなく「エンジン選定・前処理の問題」であることが判明した。
- ブラウザ配置可能な PaddleOCR系 ONNX モデル（`rapidocr-onnxruntime` で検証、Vision同様カード全体クロップ）は Vision に近い精度を再現できた（例: 「狂の魔石」conf 1.00・「3,052個」conf 0.96、「剣の秘石」はconf 0.88で「刘の秘石」と惜しい誤読）。
- アイコンのパーセプチュアルハッシュ／色ヒストグラムは、同系色ファミリー（剣/弓/槍/騎/術/殺の秘石など、色・形がほぼ同一な最頻出級の周回素材）を原理的に判別できないことを実測で確認した。この判別は名前OCRでしか解決できない。
- これらの結果は Decisions に反映済み（アイコン照合を主軸から外し、カード全体のOCRを識別の主軸に変更）。

既存資産:
- アイテムアイコン画像は `ItemIdentity` コンポーネント経由で既に配信されている（Atlas Academy 由来のアセット）。
- 全アイテムの名称は既にマスターデータ（`Item` 型の `name`）として保持されており、アイテム名OCR結果のファジーマッチ先辞書として利用できる。
- `posession` は Atlas ID をキーとした `Record<number, number>` として `localStorage` に保存され、`PossessionModal` / `MsItemsIo` から更新される。
- ビルド時にマスターデータを生成する `scripts/populate-*.ts` 群のパターンが既にある（[[project_updater_workers]]）。

## Goals / Non-Goals

**Goals:**
- 複数枚のマイルーム所持アイテムスクリーンショットから、既知アイテムの所持数候補を抽出する。
- 認識処理・画像データの送信先を**ブラウザ内に限定**し、サーバーコスト・プライバシーリスクを増やさない。
- 認識精度が不完全であることを前提に、確定前レビューで誤りを是正できる設計にする。
- iPhone（Safari）を含む主要モバイル環境で実際に動作すること（FGOはモバイル中心のゲームであり、取り込み操作自体もスマートフォンから行われる想定）。

**Non-Goals:**
- マルチモーダルLLMによる画像直読み（サーバーコストが発生するため今回は不採用。再検討条件は `docs/decisions.md` に残す）。
- サーヴァント一覧やスキル石以外の複雑なUI（強化素材以外の画面）からの取り込み。
- HTTPSスニッフィング等、ゲーム通信を利用した取り込み。
- 100%の自動認識精度の保証（レビューUIでの人手修正を前提とする）。

## Decisions

### 1. アイテム識別: カード単位OCR（名前+所持数を同時取得）を主軸、アイコン照合は補助
実測スパイクで、パーセプチュアルハッシュ／色ヒストグラムによるアイコン照合は、同系色ファミリー（剣/弓/槍/騎/術/殺の秘石など）を原理的に判別できないことが分かった。これは色・形がほぼ同一で、差はクラスを表す1文字（品目名）にしかないため。一方、カード領域（アイコン+品目名+所持数がまとまったブロック）をそのままOCRに渡すと、名前・所持数ともに高精度で読み取れることも実測できた（Context参照）。

よって識別の主軸を**カード単位のOCR結果（品目名テキスト）を既知アイテム名辞書へファジーマッチする**方式に変更する。アイコンのパーセプチュアルハッシュ照合は、OCR結果の検算・低信頼度時の補助材料として残すが、識別の必須経路ではない。
- **代替案（当初案）**: アイコンpHash候補とアイテム名OCR候補を対等に突き合わせる → 同系色ファミリーでアイコン側が実質ノイズにしかならないと実測で判明したため、名前OCRを主軸に格上げ。
- **代替案**: アイコン照合のみで識別を確定 → 同系色ファミリーを判別できず却下（変更なし）。

### 2. OCRエンジン: `ppu-paddle-ocr`（PP-OCRv6 統合多言語モデル、onnxruntime-web）を採用、tesseract.js・`@gutenye/ocr-browser` は不採用
実測スパイクで `tesseract.js`（jpn.traineddata、二値化等の前処理込み）は品目名をほぼ読み取れず、所持数も誤読が残った。一方、同じ入力に対して PaddleOCR 系 ONNX モデル（Python `rapidocr-onnxruntime` で検証）は、macOS純正 Vision framework（参照値）に近い精度を再現できた。

**実装フェーズ着手時の追加検証で重大な見落としが判明した**: Python スパイクおよび当初ブラウザ候補としていた `@gutenye/ocr-browser` は、いずれも PaddleOCR の `ch_PP-OCRv4`（中国語版）認識モデルを使用しており、その文字辞書（6623文字）にはひらがな1文字・カタカナ4文字しか含まれていない。実機ブラウザ検証（Playwright + Chromium、実カード画像3枚）でカード全体OCRは高精度（conf 0.85〜0.999）だったが、いずれのサンプルにもたまたまカタカナを含む品目名がなかったため、この欠落は当時発覚しなかった。マスターデータを確認したところ、全1928アイテム中1323件（約69%）の品目名にカタカナが含まれる（クラス名「セイバー」「アーチャー」等、多くの周回素材名）。合成テスト画像（清潔なフォントで「ホムンクルスベビー」等のカタカナ品目名を描画）で `@gutenye/ocr-browser` の `ch_PP-OCRv4` モデルを検証したところ、**検出0件**（テキスト自体を認識できない）という結果になり、この設計のままでは大多数のアイテムを識別できないことが確定した。

代替として `ppu-paddle-ocr`（TypeScript製、Node/Bun/ブラウザ/React Native横断、npm公開が数時間単位で更新される活発なメンテナンス、[GitHub](https://github.com/PT-Perkasa-Pilar-Utama/ppu-paddle-ocr)）を検証した。既定モデルの PP-OCRv6 small は「50言語以上（中国語・英語・**日本語**含む）を単一モデル・単一辞書でカバー」を謳っており、同じ合成カタカナテストで**検出・認識に成功**（conf 0.999、「ホムンクルスベビー」「セイバーピース」とも完全一致）。実カード3枚でも同等以上の精度を再現した（`card2` の「剣の秘石」は `@gutenye` 版で「刘の秘石」と誤読していたが `ppu-paddle-ocr` では正しく読めた）。

よって OCR エンジンを `ppu-paddle-ocr`（`/web` エントリ、PP-OCRv6 small モデル）に変更する。
- **代替案**: tesseract.js を第一候補のまま維持 → 実測精度が低すぎてレビューUIでの修正コストが手入力を上回りかねず、2026-06-16の意思決定ログが懸念した「本末転倒」に直結するため不採用。
- **代替案**: NDLOCR系 → 古典籍・縦書き向けで現代UIフォントとの適合度が低く、今回未検証のまま不採用継続。
- **代替案（当初案・撤回）**: `@gutenye/ocr-browser`（`ch_PP-OCRv4` モデル） → カタカナをほぼ認識できず、対象アイテムの約7割が識別不能になるため不採用。
- **[未解決・要検証]** 上記のカタカナ検証はいずれも清潔なフォント（Hiragino）で合成した画像によるもの。当初のスパイクで tesseract.js が失敗した主因はゲーム内フォント特有の縁取り・エンボス加工であり、**ゲーム内フォントで実際にレンダリングされたカタカナ品目名カードでの検証はまだ行っていない**。実機スクリーンショットでカタカナを含むカードが手元にないため、ユーザーに追加スクリーンショットを依頼して検証する（tasksに計上。この検証が完了するまで「カタカナ品目の識別」はリスクとして残る）。

### 3. 数量認識: カード単位OCRの結果から抽出（数字専用の別処理は持たない）
実測で、品目名と所持数はカード単位OCRの1回の実行で同時に読み取れることが分かったため、数字専用のテンプレート照合を別途持つ必要はない。OCR結果のテキスト行から「所持」ラベルの次に現れる `数字(,数字)*個` パターンを正規表現で抽出する。
- **代替案（当初案）**: 数字専用テンプレート照合を主軸にする → カード単位OCRで数字も十分な精度が出ており、別パイプラインを持つ複雑化のメリットがないため不採用。ただし OCR 結果の所持数が信頼度閾値を下回った場合の軽量フォールバックとして、実装フェーズで再検討の余地は残す。

### 4. アイコン・カード領域の検出: 動的グリッド検出（固定座標に依存しない）
デバイス解像度・アスペクト比の違いを吸収するため、固定ピクセル座標ではなく、画像内の輝度/エッジ検出によるセル境界検出（グリッドレイアウト推定）で**カード単位の領域**を動的に切り出す。実測でカード全体を渡した方がOCR精度が高かったため、アイコン・名前・数値を個別に細かく分割する必要はない（カード境界検出のみで足りる）。

**実装フェーズでの検証**: 実スクリーンショット（2796×1290）に対し、「カードらしい」ピクセル（R・Gが高く、B-Rが小さい＝白〜ベージュ系）の行方向の含有率をプロットしたところ、カード行と行間ギャップ（青背景のみ）が明瞭な周期パターンとして現れ、閾値ベースの行方向プロジェクション（row projection）でカード行の境界を検出できることを確認した。各行内でも同様に列方向プロジェクションで左右2カラムの境界を検出できる見込み。固定座標・機械学習モデル不要の軽量な閾値ベース手法で実現できる可能性が高い。

**実装・実画像検証で判明した3つの不具合と対処**（`lib/possession-import/detect-cards.ts`）:
1. 暗い色のアイコン（虚影の塵・宵哭きの鉄杭等）がカード内部に約9px幅の局所的な低密度ディップを作り、1枚のカードを2つに誤分割することがあった。カード間の実際の隙間（実測約34px）とは十分な差があったため、隙間の結合閾値を8px→20pxに調整して解消した。
2. 画像下端で見切れた最終行が、ピクセル境界に厳密に接していない（キャプチャの余白で数px残る）ケースがあり、単純な「バンドが画像端ピクセルに触れているか」だけでは見切れを検出できなかった。内側バンドの標準サイズに対する比率（85%未満）でも見切れと判定するロジックを追加して解消した。
3. **[最重要]** カード下部の「所持〈数量〉」欄は白背景ではなく紺色の帯になっており、当初の「カードらしい」判定（R・Gが高い＝白〜ベージュ系）はこの帯を非カードと誤判定していた。その結果、行方向プロジェクションがカード上半分（アイコン・品目名のみ）で打ち切られ、**検出された領域には所持数が一切含まれず、全アイテムの所持数candidateが常にnullになる**という重大な不具合があった。実機ブラウザ（Playwright ではなく実際にNext.js開発サーバーへ実画像をアップロードするE2E検証）で発覚し、Pythonスパイクおよびそれ以前のブラウザ内スパイク（0b/0c）はいずれもこの欠陥を検出できていなかった（手動クロップ・自動検出のクロップ範囲の両方でカード全体が十分に収まっていたため）。実測ピクセルサンプリングにより、ページ全体の背景・カード行間の隙間は「B-R（青-赤）が100以上」、カードの白背景・紺色の帯はいずれも「B-R が64以下」と明確に分離できることを確認し、判定条件を「R・Gの下限値」から「B-R の単一閾値（80以下）」に変更して解消した。

実スクリーンショット2枚（計18カード）に対し、検出→クロップ→OCRのフルパイプラインを実行し、可視カードと一致する結果を得た（0c参照）。
- **代替案**: 主要解像度ごとに座標テーブルを持つ → 新端末・新解像度が出るたびに保守が必要になるため見送り。
- **代替案（当初案）**: アイコン・名前・数値を個別領域に分割してから認識 → 実測でカード単位一括の方が精度が高く、分割は不要な複雑化と判明したため不採用。

### 5. OCR実行基盤: onnxruntime-web 最新版、COEPは導入しない（シングルスレッドWASMで十分）
当初「v1.19以降は既定ビルドがスレッド版になったため `crossOriginIsolated`（COEP）が動作要件になる」と判断したが、実際のソースコード（[backend-wasm.ts](https://github.com/microsoft/onnxruntime/blob/main/js/web/lib/backend-wasm.ts#L49-L50)、[wasm-factory.ts](https://github.com/microsoft/onnxruntime/blob/main/js/web/lib/wasm/wasm-factory.ts#L133-L149)）を確認した結果、これは誤りだった。`crossOriginIsolated` が false の場合、onnxruntime-web は `numThreads` を自動的に `1` にフォールダックし、`console.warn` を出しつつシングルスレッドで動作を継続する（例外は発生しない）。COEPは**性能を上げるオプション**であって**動作要件ではない**。

本機能はスピナー表示付きの手動一括インポート（リアルタイム性不要）であり、シングルスレッドWASMでも実用的な処理時間に収まると想定されるため、**COEPは導入せず、最新版 onnxruntime-web をそのまま採用する**。これにより、サイト全体への波及（外部リソースのCORS監査、Cloudflare Workersのレスポンスヘッダー変更、全ページの回帰確認）が不要になる。採用した `ppu-paddle-ocr` のREADMEでも同様の挙動が明記されており（`crossOriginIsolated` でなければ `numThreads` は1に固定されるだけで動作は継続する）、この判断の裏付けになった。

**追加の実装事項**: `ppu-paddle-ocr` は既定では PP-OCRv6 モデル（`.ort` 形式）を都度 `fetch()` でパッケージ提供元のCDNから取得しブラウザキャッシュに委ねる仕様。本機能は「画像はブラウザ内で処理され、外部サーバーへ送信してはならない」という要件（=送信方向の制約）には抵触しないが、可用性・オフライン耐性・自前資産管理の一貫性のため、モデル本体（検出/認識 `.onnx`）・辞書・onnxruntime-web の wasm/mjs 一式は他の静的アセット同様に自前ホスティングし、`model.detection` / `model.recognition` / `model.charactersDictionary` にローカルパスを渡す（tasksに計上）。
- **代替案（当初検討・撤回）**: 最新版 + サイト全体COEP有効化 → マルチスレッドによる性能向上が見込めるが、サイト全体への影響範囲が大きく、今回のユースケース（手動・一括・非リアルタイム）に見合わないコストと判断し撤回。将来、実機でシングルスレッド性能が明らかに不十分と分かった場合に限り、パフォーマンス改善のオプションとして再検討する。
- **代替案**: onnxruntime-web を v1.19 未満に固定 → COEP同様に不要な制約であり、最新版でも問題なく動作するため不採用。
- **[既知の懸念]** onnxruntime-web は過去に iOS Safari 16.4+ で WASM-SIMD が壊れる不具合（[microsoft/onnxruntime#15644](https://github.com/microsoft/onnxruntime/issues/15644)）を経験している。現行iOSでは解消済みとみられるが、この技術スタックはSafari固有の不具合歴があるため、実装後に**実機（iPhone Safari）での動作確認**を必須とする（tasksに計上）。

### 6. データフロー: 既存 `posession` 経路への合流
解析結果は既存の `posession`（`Record<number, number>`, Atlas ID キー）の形式にマッピングしてレビューUIに渡し、確定操作でのみ `useLocalStorage('posession', ...)` の setter を呼ぶ。新しいストレージキーやスキーマは追加しない。

## Risks / Trade-offs

- **[Risk・実装フェーズで発見・解消済み] カード検出の色閾値が所持数欄（紺色の帯）を除外してしまい、全アイテムの所持数が読み取れなくなっていた** → 実機E2E検証（Next.js開発サーバー + 実スクリーンショット）で発覚。「カードらしい」判定を R・G下限値ベースから B-R 単一閾値ベースに変更して解消（Decision 4参照）。この不具合はPythonスパイク・ブラウザ内スパイク（0b/0c）のいずれでも検出できておらず、**実際のアプリケーションへの統合・実データでのE2E検証が、個別のライブラリ検証だけでは代替できない**ことを示す教訓となった。
- **[Risk] ゲームUIの表示変更でアイコン/OCRモデルが陳腐化する** → アイコン照合は既存の保守モデル（マスターデータ更新ジョブ、[[project_updater_workers]]）を踏襲。OCRモデル自体は既知アイテム名辞書へのファジーマッチで吸収されるため、UI文言の細かな変更には比較的強い。
- **[Risk] 端末・OS・スクリーンショット圧縮方式による画質差で誤認識が増える** → 確定前レビューUIを必須化し、誤り修正・除外を常に可能にする（specs 側で SHALL NOT 自動確定と規定済み）。
- **[Risk] 動的グリッド検出が失敗する异常なレイアウト（分割画面、余計なUI要素混入等）** → 検出0件時はエラーではなく「認識できたアイテムがありません」として通常の手入力へ誘導する。
- **[Risk] スパイクの精度実測は手動で微調整したクロップに基づくものであり、自動のグリッド検出が同等品質のカード切り出しを再現できるかは未検証** → 実装フェーズで自動検出結果のクロップ品質・OCR精度を実測し、スパイクの数値（conf 0.88〜1.00等）が再現するかを確認するタスクを設ける（tasksに計上）。アイコンpHash照合が当初うまくいかなかった一因も、クロップの粗さにあった可能性が高い。
- **[Risk] 複数スクリーンショットの継ぎ目でカードが部分的にしか写らない**（スクロール境界での見切れ、次の画像との重複範囲）→ 「3,052」が「3,05」のように途中で切れて“もっともらしいが誤った”所持数として読み取られる恐れがある。画像端に接するカード（上端/下端が画像境界で切れている）は低信頼または除外扱いとし、他の画像との重複領域は同一アイテムとして統合ロジック（specs「複数画像間での重複・矛盾の統合」）に委ねる。仕様・実装の両方で明示的に扱う（tasksに計上）。
- **[Risk] ファジーマッチは同系色ファミリー（剣/弓/槍/騎/術/殺の秘石等）の誤読を救えない** → これらの品目名は互いに編集距離1程度しか離れておらず、ファジーマッチは誤読を別の実在アイテム名に「もっともらしく」寄せてしまう（訂正ではなく別の誤りにすり替わる）。実際の安全網は**信頼度閾値によるOCR結果そのものの「要確認」振り分け**であり、ファジーマッチによる誤り訂正ではない。閾値のチューニングをsafety-criticalな作業として扱う。
- **[Risk・実装フェーズで確認済み] 実際にブラウザへ組み込むOCRパッケージが、スパイクで検証したモデルと異なる可能性** → 実際に的中した。Python版 `rapidocr-onnxruntime` と当初のブラウザ候補 `@gutenye/ocr-browser` はいずれも `ch_PP-OCRv4`（中国語版、カタカナをほぼ含まない辞書）を使っており、カタカナを含む品目名（全体の約69%）を識別できないことが判明した。`ppu-paddle-ocr`（PP-OCRv6統合多言語モデル）に切り替えて解決（Decision 2参照）。
- **[Risk・未解決] ゲーム内フォントでレンダリングされたカタカナ品目名の実機検証がまだない** → `ppu-paddle-ocr` のカタカナ認識は清潔なフォント（Hiragino）で合成した画像でのみ確認済み。当初 tesseract.js が失敗した主因はゲーム内フォント特有の縁取り・エンボス加工であり、フォントスタイルが精度に影響することは実証済み（Context参照）。手元のサンプルスクリーンショットにカタカナを含むカードが写っていないため、ユーザーから追加スクリーンショットを入手し、ゲーム内フォントでのカタカナ認識精度を検証するまでは未解決のリスクとして残す（tasksに計上）。
- **[Risk] OCRが桁区切りのカンマをピリオドと誤読することがある**（実測: 「3,052個」→「3.052個」）→ 所持数のパース処理は `,` と `.` の両方を桁区切りとして正規化してから数値化する（`3.052` を `3` と誤解釈しない）。
- **[Risk] onnxruntime-web はSafari/iOS向けにWASM-SIMD破損の不具合歴がある**（[#15644](https://github.com/microsoft/onnxruntime/issues/15644)）→ 実装後に実機（iPhone Safari）での動作確認を必須タスクとする。問題が再発した場合はバージョン固定やフォールバック（他エンジンへの切替、機能の一時無効化）を検討する。
- **[Trade-off] シングルスレッドWASMは複数枚一括投稿時に処理時間が伸びる可能性がある** → 手動・非リアルタイムのインポートフローであるため許容範囲と想定するが、実機検証で明らかに実用に耐えない場合は、COEP有効化によるマルチスレッド化を改めて検討する（Decision 5 参照。ただしサイト全体への影響範囲が大きいため安易に採用しない）。

## Migration Plan

- 新規機能のため既存データへの移行は不要。既存の `posession` 手入力・JSON貼り付け経路は変更せず併存させる。
- COEPは導入しないため、既存ページへの影響やヘッダー変更のロールバック計画は不要。
- ロールバックは、スクリーンショット取り込みUIの導線（エントリポイント）を非表示にするだけで完了する（`posession` のスキーマに変更がないため）。
- リリース後、`docs/decisions.md` の 2026-06-16 エントリに本 change の実施内容と結果（精度実測値等）を追記し、意思決定ログを更新する。

## Open Questions

- カード単位OCR（PaddleOCR系ONNX + ファジーマッチ）の実測精度が、レビューUIでの修正コストを手入力より下げられるか（実装後にユーザーの手直し件数で検証する）。
- 自動グリッド検出によるカード切り出し品質が、スパイクで手動クロップした場合の精度（conf 0.88〜1.00）にどこまで近づけるか。
- カード領域のグリッド検出アルゴリズムの具体的な閾値・パラメータ、および画像端で部分的に切れたカードの除外基準は実装フェーズで調整する。
- アイテム名のファジーマッチ類似度閾値、および「要確認」に振り分ける信頼度閾値は実装フェーズでサンプル画像を用いて調整する。
- 実際に採用するブラウザ向けOCRパッケージ（`@gutenye/ocr-browser` 等）の精度が、スパイクで検証した `rapidocr-onnxruntime`（PP-OCRv4系）と同等か。
- シングルスレッドWASMでの複数枚一括処理時間が実機（特にiPhone）で許容範囲に収まるか。収まらない場合、Decision 5 のCOEP案を改めて検討する。
