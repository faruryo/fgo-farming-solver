# Design

## 確定済みの製品判断（ユーザー決定）

1. **聖晶石→課金額の換算レート**: App Store の**最大金額パックの単価**を採用（最大パックが最安単価）。裏取り済み現行値 = **¥10,000 / 168個 ≒ ¥59.5/個**。階段最適化（分割・福袋・セール）はしない。換算式はインフォメーションで明示する。
2. **AP回復はレベル依存**: `金果実=40AP / 銀林檎=20AP` 固定は廃止。マスターレベル → 最大AP テーブルを基準に、聖晶石・黄金の果実＝最大AP（全回復）として扱う。
3. **マスターレベルはクラウド同期**: `use-cloud-sync` の `KEYS` に `masterLevel` を追加（横断利用のため）。未入力時は**最大レベル既定**。
4. **所持りんごは同期しない**: 消費で頻繁に変わるため localStorage 端末ローカル永続のみ（`KEYS` に追加しない）。
5. **終了済みイベントもシミュレート可**: テスト都合（会期ハードコード）の解消も兼ねる。

## 状態と同期のアーキテクチャ

`use-cloud-sync.ts` の `KEYS` 配列に列挙された localStorage キーが自動でクラウド同期される（`material` 等）。この仕組みをそのまま使う:

- `masterLevel`（新規 localStorage キー、`number`）→ **`KEYS` に追加** ＝ 自動でクラウド同期。
- りんご所持数（例 `events/apples` キー、`Record<appleType, number>`）→ localStorage に保存するが **`KEYS` には追加しない** ＝ 端末ローカル永続のみ。

> `ChaldeaState`（`hooks/create-chaldea-state.ts`）はサーヴァントID をキーにした map（`{ [id]: ServantState }`）なので、スカラーの `masterLevel` をここに混ぜない。独立した localStorage キーにする。

### 未入力時の既定（最大レベル）

`masterLevel` が未設定（`null`/未同期）のときは、テーブルの最大レベル（現行 200）の最大AP を用いる。明示入力があればそれを優先。

## AP回復モデル

```
maxAp = maxApTable[masterLevel]            // レベル依存
needAp = solverResult.totalAp              // 計画の必要AP（既存ソルバ出力）

// 所持りんごを優先消費（回復量の大きい順など、消費順は要определ）
for apple in ownedApples (consume order):
    recovered = apple.recoverAp(maxAp)     // 全回復系は maxAp、固定系は定数
    consume min(owned, ceil(remaining/recovered))
    ...

// 残った needAp を聖晶石で賄う（聖晶石1個 = maxAp 全回復）
quartzCount = ceil(remainingAp / maxAp)
yen = quartzCount * QUARTZ_UNIT_YEN        // QUARTZ_UNIT_YEN = 10000/168
```

### 要確定データ（実装前に出典で固める）

- **D1: マスターレベル → 最大AP テーブル（1〜現行最大）**。Gamerch 早見表等で確定。既知の足がかり: ML170→146, ML180→148前後, 正月2026で上限200開放。
- **D2: 各 AP 回復アイテムの回復量ルール**。聖晶石・黄金の果実＝最大AP（全回復, レベル依存）。白銀/青銅の果実が「固定量」か「最大APの割合」かを出典で確認（"レベルに応じて回復量が違う" 要件を満たすため、各アイテムが全回復系か固定系かを正しく分類する）。
- **D3: 聖晶石単価定数**。現行 ¥10,000/168個。値は将来の価格改定で変わりうるため定数として一箇所に置く。

## インフォメーション文言（換算式の明示）

計画ページの金額表示の近くにツールチップ/注記を置く。例（ja）:

> 課金額は「不足AP → 聖晶石（1個でAP全回復）に換算 → App Store 最大パック単価（¥10,000 / 168個）で金額化」で算出した目安です。実際の最安価格は少額パックの分割購入・福袋・セールにより変動します。

## 終了済みイベントのシミュレート

ソルバ・計画ロジックは会期に依存しないため、一覧の選択可能対象に終了済みを含めるだけで成立する。終了済みは会期外バッジで明示し、計画ページ自体は開催中と同一 UI。

## スコープ分担

- 本 change: AP内訳・聖晶石換算・課金額・マスターレベル同期・終了済みシミュレート。
- `event-lottery-planner`（別 change）: `/events` ナビ導線（オーファン解消）・アイテム名解決。本 change はそれらの完了を前提に積む。
