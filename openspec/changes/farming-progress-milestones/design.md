## Context

現状、ユーザーが素材所持数を更新（同期）した際、裏側で `farming_results` として履歴が保存されていますが、ユーザーには「更新完了」の通知のみが届きます。
この「更新」というアクションを、「努力の成果の確認」というポジティブな体験に変換するための設計を行います。

## Goals / Non-Goals

**Goals:**
- 過去の特定時点（前回、1週間前、1ヶ月前）とのデータ比較ロジックの確立。
- 達成感を感じさせる「進捗報告」UIの設計。
- お気に入りサーヴァントからのフィードバック（褒め言葉）の実装。

**Non-Goals:**
- 周回中のリアルタイムな素材獲得トラッキング（あくまで同期・更新時の集計）。
- ソーシャル共有機能の詳細実装（将来的な拡張要素とする）。

## Decisions

1. **比較データの取得方法**
   - **決定**: D1の `farming_results` テーブルから、`updatedAt` を基準に「前回」「約7日前」「約30日前」のレコードを取得する。
   - **理由**: すでに履歴データが保存されているため、追加のデータモデル変更なしに実装可能。

2. **進捗の定義**
   - **決定**: 「目標達成に必要な合計素材数の減少」および「個別の不足素材の解消」を進捗として扱う。
   - **理由**: ユーザーが最も関心があるのは「あとどれだけ周回すればいいか」の減少であるため。

3. **サーヴァントフィードバックの演出**
   - **決定**: shadcn/ui の Dialog (Modal) または Drawer を使用し、サーヴァントの画像とセリフを表示する。
   - **理由**: モバイル・デスクトップ両方で視覚的に目立ち、達成感を強調できるため。

4. **お気に入りサーヴァントの特定**
   - **決定**: `objective` リストの先頭のサーヴァント、または将来的に追加される「お気に入り設定」を使用。設定がない場合はマシュ（デフォルト）を表示。
   - **理由**: ユーザーが現在育成しようとしているサーヴァントに褒められるのが最も効果的。

## Risks / Trade-offs

- [Risk] 過去データが存在しない新規ユーザーの場合、比較ができない。 → [Mitigation] 「最初のデータ登録完了！」といった専用の歓迎メッセージを表示する。
- [Risk] 大量に素材を消費した（育成を進めた）直後は「所持数」が減るため、単純な所持数比較では進捗が分かりにくい。 → [Mitigation] 「必要数（残り）」の減少をメインの指標とする。
