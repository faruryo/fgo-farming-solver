## Context

現状、ユーザーが素材所持数を更新（保存）した際、裏側で `farming_results` として履歴が保存されていますが、ユーザーには「更新完了」の通知のみが届きます。
この「更新」というアクションを、「努力の成果の確認」というポジティブな体験に変換するための設計を行います。

## Goals / Non-Goals

**Goals:**
- 過去の特定時点（前回、1週間前、1ヶ月前）とのデータ比較ロジックの確立。
- 達成感を感じさせる「進捗報告」UIの設計。
- マシュからのフィードバック（褒め言葉）の実装。
- 新規入手サーヴァントによる必要AP増加を進捗判定から正しく差し引く。
- 高難易度コンテンツ（オーディールコール、冠位戦）へのアクセス可否を考慮した進捗評価。

**Non-Goals:**
- 周回中のリアルタイムな素材獲得トラッキング（あくまで保存・更新時の集計）。
- ソーシャル共有機能の詳細実装（将来的な拡張要素とする）。
- アニメーション演出（v1 では色とサイズのみで達成感を表現）。
- マシュ以外のサーヴァントによるフィードバック（v2 以降の拡張）。

## Decisions

1. **比較データの取得方法**
   - **決定**: D1 に新規テーブル `state_snapshots` を作成し、cloud save 時と solver 実行時にユーザーの localStorage 全体を JSON snapshot として保存する。同日内の重複は UPDATE で1日1件に制限する（同日上書き戦略）。
   - **理由**: auto-sync 5秒デバウンスでの保存頻度をそのまま履歴に流すと容量が爆発するが、1日1件に間引けば「先週」「先月」比較に十分。既存の `farming_results` テーブルは solver の結果保存に特化しており、「ユーザー状態全体」を保存する用途とは責務が異なる。

2. **進捗指標**
   - **決定**: 以下3指標を「変化があったものだけ表示」する方針で運用する。
     - `total_ap` の減少（新規入手サーヴァント分をオフセットした純粋値）
     - サーヴァント成長度（chaldea state のスキル+アペンドレベル合計の増加）
     - 目標 `total_ap` の増加（「より高い目標を掲げた」とポジティブ表示）
   - **理由**: 単純な所持数比較は素材消費（スキル上げ等）で簡単にマイナスになり、進捗の演出として適さない。「進んだ部分だけ目立たせる」設計により常にポジティブな体験を提供する。

3. **AP進捗の tier 判定**
   - **決定**: 経過時間から自然回復AP（5分=1AP）を算出し、`deltaAp` との比で 4 段階に分類する。
     ```ts
     const naturalRecoveryAp = elapsedMinutes / 5
     if (deltaAp >= naturalRecoveryAp * 1.5) tier = 'large'
     else if (deltaAp >= naturalRecoveryAp)  tier = 'medium'
     else if (deltaAp > 0)                    tier = 'small'
     else                                     tier = 'none'
     ```
   - **理由**: 比率ベースの閾値は所持サーヴァント数が増えると破綻するが、自然回復AP基準なら経過時間に対する「相対的な頑張り度合い」を測れる。長期間ログインしないユーザーでも公平に評価され、イベント期間の高効率周回も `large` として正しく評価される。

4. **新規入手サーヴァントによる必要AP増加のオフセット**
   - **決定**: 比較スナップショット間で「新規 owned 化された」サーヴァント（`disabled: true` → `false`）を検出し、rarity 別の事前計算済み推定APを `deltaAp` に足し戻す。
   - **理由**: ガチャで新サーヴァントを引いた直後は必要APが増えて見え、純粋な周回成果を測れなくなる。rarity 別オフセットで「サーヴァント獲得を進捗の阻害要因に見せない」ことが目的。
   - **rarity 別 AP 推定値の算出**:
     - 各 rarity（1〜5）から代表サーヴァントをランダム選択（複数体）。
     - そのサーヴァントを「素体（disabled）」から「育成完了（共通目標達成）」にするための必要APを solver で計算。
     - 複数サンプルの平均または中央値を採用。
     - 高難易度アクセス可否で 2 セット（基本 / 高難易度あり）を生成。

5. **高難易度コンテンツアクセス可否の判定**
   - **決定**: ユーザーの localStorage `quests`（`checkedQuests`）に「オーディールコール / 冠位戦」の代表クエストIDが含まれているかで推測する。明示設定UIは v1 では追加しない。
   - **理由**: 既存のクエスト選択UIから自然に検出できるため新規入力UIが不要。誤検出した場合もユーザーがチェックON/OFFすることで自己修正可能。将来的に明示設定への切り替えは可能。

6. **マシュ固定 (v1)**
   - **決定**: 進捗演出のサーヴァント画像とメッセージは「マシュ・キリエライト」固定。マシュメッセージ辞書は [appmedia の引用リスト](https://appmedia.jp/fategrandorder/75727618) を参考に口調を統一する。
   - **理由**: サーヴァントごとに口調・性格が大きく異なるため、汎用メッセージを使い回すと没個性になる。マシュは導入から共通の伴侶キャラで違和感が小さく、敬語＋「先輩」呼びの一貫したトーンで全進捗種別をカバーできる。将来サーヴァント追加は「サーヴァントID × メッセージ辞書」の形式で拡張可能。

7. **進捗ゼロ・比較対象不在時の振る舞い**
   - **決定**: 以下のいずれかに該当する場合は「登録お疲れ様」モードで表示する。
     - 過去スナップショットが存在しない（初回ユーザー）。
     - 指定期間（前回 / 1週間前 / 1ヶ月前）のスナップショットが存在しない。
     - 全指標で改善ゼロ。
   - **理由**: モーダルを抑制すると「保存できたのかどうかわからない」体験になる。完了確認を兼ねた「お疲れ様」表示で UX を一貫させる。

8. **演出は色・サイズで達成感を表現（アニメーション無し）**
   - **決定**: tier ごとに背景色・テキストサイズ・装飾の強度を変える（`large` は派手な配色＋大きい数値、`small` は控えめ）。アニメーションは v1 では実装しない。
   - **理由**: 静的な視覚差でも tier の区別は十分つけられる。アニメーション実装はコストが高く、視覚的にうるさくなりやすい。必要なら v2 で追加。

## Risks / Trade-offs

- [Risk] **rarity 別 AP 事前計算が古くなる**（マスターデータ更新でドロップ率や必要素材が変わる） → [Mitigation] master-data 更新時に再計算をトリガーするか、ビルド時に毎回計算する。
- [Risk] **`state_snapshots` の容量増加** → [Mitigation] 同日上書きで1ユーザー1日1件に制限し、さらに古い snapshot は将来的に cron で間引く保守スクリプトを追加可能とする。
- [Risk] **過去データが存在しない新規ユーザーの場合、比較ができない** → [Mitigation] 「最初のデータ登録完了！」専用の歓迎メッセージで体験を保護。
- [Risk] **高難易度アクセスの推測誤り** → [Mitigation] `checkedQuests` 推測の誤検出時はユーザーがチェック状態を変えて自己修正可能。将来は明示設定UIに切り替え可能な設計とする。
- [Risk] **大量に素材を消費した直後は「所持数」が減るため、単純な所持数比較では進捗が分かりにくい** → [Mitigation] 「必要数（残り）」の減少を主指標とし、所持数の生比較はサマリー表示から除外。
- [Risk] **マシュ口調メッセージのバリエーション不足で繰り返し感が出る** → [Mitigation] tier × 進捗種別の組み合わせごとに最低 3〜5 種の文言バリエーションを用意し、ランダム選択。
